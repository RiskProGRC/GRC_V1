Change 1 — AJAX: Remove redundant beforeSend
Removed redundant beforeSend callback
assets/js/usermanagement.js — .adduser-button click handler
Format

The beforeSend callback re-ran field emptiness checks after GRC.validate() had already validated the form before $.ajax was called. login.js does all validation inline before calling $.ajax — no beforeSend is needed.

Before
beforeSend: function () {
  var empty = addUserRules.some(...);
  if (empty) {
    $btn.prop('disabled', false);
    showFeedback('...', 'warning');
    return false;
  }
},
After
// removed entirely —
// GRC.validate() already
// guards this path
Change 2 — AJAX: Centralise button re-enable in complete:
Moved button re-enable from success/error into a single complete:
assets/js/usermanagement.js — .adduser-button click handler
Format

login.js uses one complete: callback to re-enable the submit button, guaranteeing it fires regardless of success or failure. The old code repeated $btn.prop('disabled', false) inside both success and error.

Before
success: function (res) {
  ...
  $btn.prop('disabled', false).text('ADD USER');
},
error: function () {
  ...
  $btn.prop('disabled', false).text('ADD USER');
}
After
complete: function () {
  $btn.prop('disabled', false).text('ADD USER');
},
success: function (res) { ... },
error: function () { ... }

Change 4 — Swal: shorthand replaced with object syntax
Converted all Swal.fire(title, text, icon) shorthand to named-property object syntax
assets/js/usermanagement.js — access-toggle, delete, reset-pw handlers
Format

login.js always uses the object form Swal.fire({ icon, title, text }). usermanagement.js was using the 3-argument positional shorthand in 7 places, which is less readable and harder to extend with additional options (e.g. timer, confirmButtonText).

7 calls updated across the access-toggle, delete-user, and reset-password handlers:

Before (shorthand)
Swal.fire('Error', res.message, 'error');

Swal.fire('Deleted', name + '...', 'success');

Swal.fire(
  ok ? 'Done' : 'Error',
  res.message,
  ok ? 'success' : 'error'
);
After (object syntax)
Swal.fire({ icon: 'error', title: 'Error', text: res.message });

Swal.fire({ icon: 'success', title: 'Deleted', text: name + '...' });

Swal.fire({
  icon: ok ? 'success' : 'error',
  title: ok ? 'Done' : 'Error',
  text: res.message
});
No visual change to the user — SweetAlert2 renders identically from either form. This is a code-consistency change only.
Summary
What was not changed
Note
  • showFeedback() on the add-user call was left in place — it targets a #form-feedback div in the HTML and replacing it with Swal.fire() requires a separate HTML-side decision.
  • The timer: property was not added to delete/reset-pw success Swals because neither is followed by a page redirect — a timer would dismiss the confirmation before the user reads it.
  • The response vs res parameter name difference was left — it is local to each callback and has no cross-file impact.
  • No backend endpoints or PHP files were touched.
