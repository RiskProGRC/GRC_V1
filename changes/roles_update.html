Role System Redesign

Replacing permission-table nav guards with role-based checks. 5 files changed, permission table kept in database but no longer drives the nav or any UI element.

📋

Overview

What is changing and why

Currently the nav menu visibility is controlled by a separate permission table — each user gets a row with 19 boolean columns. This is being replaced with a simpler 3-role system where the role number on the user record determines what they can see.

The data scoping in header.php (dept-only vs all data) already works on roles and stays untouched in logic — only a 3rd branch for Manager is added.

New role definitions

ValueLabelStatus
1AdministratorUnchanged
2Champion (was: User)Renamed
3ManagerNew

No database migration needed — role values 1 and 2 are unchanged; 3 is a new integer stored in users.roles.

Change 2 — header.php

layout/header.php   lines 74–175 (all data fetch blocks)

Add role 3 (Manager) to every data-scoping block

Currently each block has if roles==1 … elseif roles==2. Manager (3) gets the same dept-scoped fetch as Champion (2). The cleanest change is to collapse both into an else.

// Current pattern (repeated ~9 times): - if($sess_roles==1){ - $showrisk=$riskClass->showRisk(); - }elseif($sess_roles==2){ - $showrisk=$riskClass->showRiskdept($sdid); - } // New pattern: + if($sess_roles==1){ + $showrisk=$riskClass->showRisk(); + }else{ + $showrisk=$riskClass->showRiskdept($sdid); + }

This covers roles 2 (Champion) and 3 (Manager) in a single else. Any future role added will default to dept-scoped, which is the safe/restrictive default.
Affected blocks: Department, Process, Risk, Risk Category, Control, Control Strength, KI, Action, Incident, Recommend, KRI — 11 blocks total.
