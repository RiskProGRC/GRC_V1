Critical

1 — Session Fixation: Regenerate Session ID After Login New

Pending

File: Project/login/loginClass.php — login() method

After a successful login the session ID is kept from before authentication. An attacker who tricks a victim into using a known session ID (session fixation) will have full access once that session becomes authenticated. One line fixes this.

Why neither report caught this: It's invisible in the code — the session is set correctly but the ID is never rotated. It looks fine until you think about what the session ID was before the user logged in.
Before — session ID unchanged after login
$_SESSION['uid']     = $row['id'];
$_SESSION['user']    = $row['username'];
$_SESSION['dept_id'] = $row['dept_id'];
$_SESSION['roles']   = $row['roles'];
// session ID is still whatever it was before login
After — new session ID issued on every login
session_regenerate_id(true); // rotate session ID — invalidates old ID immediately
$_SESSION['uid']     = $row['id'];
$_SESSION['user']    = $row['username'];
$_SESSION['dept_id'] = $row['dept_id'];
$_SESSION['roles']   = $row['roles'];
Effect: Any session ID an attacker knew before login becomes invalid the moment the user authenticates. Session fixation attacks are eliminated.
Critical

2 — Debug Log Writes SQL Queries and Emails to a Public File New

Pending

File: Project/login/loginClass.php — googlesignup() — lines 58–163

googlesignup() opens a file called googlesignup_debug.log and writes the full SQL INSERT statement (including the hashed password), the email address, and session data to it on every Google login. This file sits in the web root — anyone who knows its name can read every Google user's email and session data.

Why this is critical: Debug logging was useful during development but was never removed. A public log file containing user emails and SQL queries is a data breach waiting to happen.
Before — 15+ lines writing to a public log file
$debug_file = fopen("googlesignup_debug.log", "a");
fwrite($debug_file, "SQL Query: $insert\n");     // raw SQL with password hash
fwrite($debug_file, "Session set: uid=...\n");   // session data
fwrite($debug_file, "Checking email: $email\n"); // user PII
...
After — all debug lines removed
// No debug logging. Use PHP error_log() to the server error log
// if temporary debugging is needed, never a public web-root file.
Effect: User emails, SQL, and session data are no longer written to a publicly accessible file. The googlesignup method becomes clean and safe.
Critical

3 — Reset Token Has No Expiry — Old Links Work Forever New

Pending

Files: reset-password-action.php   Project/login/loginClass.php — passupdate()

When a user requests a password reset, a token is stored in the database with no timestamp. If the email is intercepted or the link is found in browser history six months later, it still works. Password reset tokens should expire within 30–60 minutes.

My recommended solution: Add a token_expires_at DATETIME column to the users table. Set it to NOW() + 1 hour when the token is saved. Check it in passupdate() before allowing the reset.
Recommended — DB change
ALTER TABLE users ADD COLUMN token_expires_at DATETIME NULL AFTER verify_token;
reset-password-action.php — set expiry when saving token
$token   = bin2hex(random_bytes(16)); // secure token (replaces md5(rand()))
$expires = date('Y-m-d H:i:s', strtotime('+1 hour'));

$stmt = $loginClass->saveResetToken($email, $token, $expires); // new method
loginClass passupdate() — check expiry before accepting token
$row = $this->fetchOne('users', 'verify_token', $token);
if (!$row)                                    { return 'Invalid token'; }
if (strtotime($row['token_expires_at']) < time()) { return 'Token expired. Request a new reset link.'; }
Effect: Reset links are valid for 1 hour only. Intercepted or old links are automatically rejected.

4 — XSS on confirmpass.php GET Values

Pending

File: confirmpass.php — lines 82, 86

$_GET["token"] and $_GET["email"] are printed into HTML attributes without escaping. A crafted URL injects JavaScript into the page.

Before
echo $_GET["token"];  // XSS
echo $_GET["email"];  // XSS
After
echo htmlspecialchars($_GET["token"], ENT_QUOTES, 'UTF-8');
echo htmlspecialchars($_GET["email"], ENT_QUOTES, 'UTF-8');
Effect: Script injection via the password reset URL is blocked.

5 — uid Taken from Browser (Hidden Field) in confirmaction.php

Pending

File: confirmaction.php — line 6

The uid from a hidden form field is trusted. Any user can change it in DevTools and update someone else's password.

Before
$uid = $_POST["uid"]; // browser-controlled
After
session_start();
if (!isset($_SESSION['uid'])) { Response::error('Session expired'); }
// uid always comes from session — never from the form
Effect: Password update is bound to the active session. No user can update another account's password.
Security + New

6 — No CSRF Token on Login and Reset Forms New

Pending

Files: login.php   reset-password.php   loginaction.php

Neither the login nor the password-reset form has a CSRF token. A malicious page on another domain could submit these forms on behalf of a logged-in user. My recommended fix generates a token in the session and verifies it server-side.

Why this was missed: Both reports focused on the class and handlers. CSRF lives on the form HTML and the very first line of the handler — easy to overlook.
Recommended — generate token in login.php
<?php
session_start();
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}
?>
...
<form id="loginform">
    <input type="hidden" name="_token" value="<?= $_SESSION['csrf_token'] ?>">
    ...
Recommended — verify in loginaction.php
session_start();
if (!isset($_POST['_token']) || $_POST['_token'] !== $_SESSION['csrf_token']) {
    Response::error('Invalid request. Please refresh the page and try again.');
}
Effect: Cross-site form submissions are rejected. The token rotates on each page load so it cannot be reused.

7 — Email Enumeration: Reset Form Reveals If an Account Exists New

Pending

File: reset-password-action.php — lines 76–80

When the email is not found the server returns "ERROR CHECK"; when it is found it returns "its available". This lets an attacker enumerate which email addresses have accounts by submitting the form in bulk. The fix is to always return the same message regardless of whether the email exists.

Before — different responses reveal account existence
if(mysqli_num_rows($query_check) > 0){
    ...
    echo "its available";   // ← account exists
} else {
    echo "ERROR CHECK ";    // ← no account
}
After — identical response either way
// Always respond with the same message.
// If the email exists, send the reset link silently.
// If it doesn't, do nothing silently.
Response::success('If that email is registered, a reset link has been sent.');
Effect: An attacker cannot tell from the response whether an email address has an account. The message is truthful either way.
Security

8 — Brute Force: No Rate Limiting on Login Attempts

Pending

File: Project/login/loginClass.php — login() method

The old commented-out login() tracked attempts in login_attempt table but never enforced a lockout. My recommendation: track by both IP and email, and lock after 5 failures within 15 minutes.

Recommended — track per IP and per email
private function countRecentFailures(string $column, string $value): int {
    $window = date('Y-m-d H:i:s', strtotime('-15 minutes'));
    $stmt   = $this->prepare(
        "SELECT COUNT(*) FROM login_attempt WHERE `$column`=? AND attempted_at > ?"
    );
    $stmt->bind_param('ss', $value, $window);
    $stmt->execute();
    return (int) $stmt->get_result()->fetch_row()[0];
}

// In login() — check before even querying for the user:
if ($this->countRecentFailures('ip_address', $ip) >= 5 ||
    $this->countRecentFailures('email', $email) >= 5) {
    return 'locked'; // UI shows: "Too many attempts. Try again in 15 minutes."
}
Effect: Password guessing is capped at 5 attempts per 15 minutes, measured per IP AND per email so VPN-hopping doesn't bypass the check.
Security

10 — isset() Guards on All $_POST Reads in Handler Files

Pending

Files: loginaction.php   confirmaction.php   reset-password-action.php

All three handlers read $_POST keys directly without isset(). A tampered request that omits a key causes PHP notices and undefined variable passed into the database layer.

Pattern to apply to every handler
if (!isset($_POST['email'], $_POST['password'])) {
    Response::error('Missing required fields');
}
$email = $_POST['email'];
$pass  = $_POST['password'];
Effect: Malformed or incomplete POST requests exit cleanly with a JSON error rather than causing undefined variable warnings.

11 — Submit Button Has No Loading State (Double-Click Risk) New

Pending

File: login.php — JS

Clicking "Log in" twice before the AJAX returns fires two simultaneous login requests. The button should be disabled during the request and re-enabled on completion. This also gives the user a visual cue that something is happening.

Recommended — disable on click, re-enable on response
var $btn = $('.login-btn');
$btn.prop('disabled', true).text('Logging in...');

$.ajax({
    ...
    complete: function() {
        $btn.prop('disabled', false).text('Log in'); // always re-enable
    }
});
Effect: No duplicate login requests. User sees immediate feedback that the click was received.

12 — Wrong Credentials: Stay on Page, Clear Error Message, No Redirect

Pending

File: login.php — lines 129–137

Currently redirects back to login.php on wrong credentials, clearing the email field. The SweetAlert title says "login unsuccess". My fix: show the error inline without any page reload, with a clear English message.

Before
Swal.fire({ icon:"error", title:"login unsuccess", timer:3000 })
window.location.href = "login.php"; // clears the form
After
Swal.fire({
    icon: 'error',
    title: 'Login Failed',
    text: response.message, // "Invalid email or password. Please try again."
    confirmButtonText: 'Try Again'
    // no redirect — email field stays filled in
});
Effect: User does not lose their email address on a mistyped password. Clear grammar. No unnecessary page reload.

14 — No autocomplete Attributes on Login Fields New

Pending

File: login.php — lines 54, 60

The email and password fields have no autocomplete attributes. Browsers guess the type and sometimes fill the wrong values. Explicit attributes also allow password managers to fill in credentials correctly.

Before
<input type="text"     name="email"    ...>
<input type="password" name="password" ...>
After
<input type="email"    name="email"    autocomplete="email"            ...>
<input type="password" name="password" autocomplete="current-password"  ...>
Effect: Password managers work correctly. Browsers fill the right fields. type="email" also adds mobile keyboard optimisation and basic format checking.

16 — systemlog() Has a Logic Bug: count < 0 Can Never Be True New

Pending

File: Project/login/loginClass.php — lines 340–357

num_rows is always ≥ 0, so the condition if($count < 0) can never be true. The "NULL VALUES" return path is dead code. The method also uses an unnecessary while loop when fetchAll() does the same thing in one line.

Before — dead branch + manual loop
$count = $query->num_rows;
if($count < 0){            // NEVER TRUE — num_rows is always >= 0
    return "NULL VALUES";
}else{
    while($row = $query->fetch_assoc()){
        $rows[] = $row;
    }
    return !empty($rows) ? $rows : array();
}
After — ARIA fetchAll, one line
public function systemlog(): array {
    return $this->fetchAll('system_logs'); // returns [] if empty, never null
}
Effect: Dead code removed. Consistent with every other showX() method in the system after ARIA Phase 2.

18 — Move Login JS to assets/js/login.js (ARIA Phase 4)

Pending

Files: login.php   reset-password.php   confirmpass.php

All three view files have large inline <script> blocks. Following ARIA Phase 4, move all login-related JS into a single assets/js/login.js file. PHP values the JS needs (CSRF token, page flags) are passed via a small inline config object.

Recommended — pass config from PHP, load from file
<script>
window.GRC_LOGIN = {
    csrfToken: '<?= $_SESSION['csrf_token'] ?>',
    confirmpassUrl: 'confirmpass.php',
    systemoverUrl:  'systemover.php'
};
</script>
<script src="assets/js/login.js"></script>
Effect: PHP view files contain only HTML. All JS is in one cacheable file. Consistent with how the rest of the system works after ARIA Phase 4.
