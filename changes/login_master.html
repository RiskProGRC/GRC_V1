Step 1 — Critical Stops
7 issues that cause data loss, account takeover, or completely broken features. Fix these before anything else — none of the subsequent steps should be done while these exist.
Critical New Finding

1.1 — Session Fixation: No session_regenerate_id() After Login

Done
New FindingProject/login/loginClass.php — Login_test() line 250

After a successful password check the session ID is never rotated. An attacker who pre-plants a known session ID on the victim's browser (session fixation) will be fully authenticated the moment the victim logs in.

Before — session ID unchanged
if (password_verify($pass, $row['upassword']) && $row["user_type"] == 1) {
    $_SESSION["uid"]     = $row["id"];
    $_SESSION["user"]    = $row["username"];  // old session ID still active
After — one line closes session fixation
if (password_verify($pass, $row['upassword']) && $row["user_type"] == 1) {
    session_regenerate_id(true); // issue a new ID, destroy the old one
    $_SESSION["uid"]     = $row["id"];
    $_SESSION["user"]    = $row["username"];
Effect: Any session ID the attacker knew before login is invalidated the moment the user authenticates.
Critical New Finding

1.2 — Debug Log Writes SQL + Emails to a Public File on Every Google Login

Done
New FindingProject/login/loginClass.php — googlesignup() lines 58–163

googlesignup() opens googlesignup_debug.log in the web root and writes user emails, raw SQL (including hashed passwords), and session data to it on every call. This file is publicly readable by anyone who knows its URL.

Before — live data written to public file
$debug_file = fopen("googlesignup_debug.log", "a");
fwrite($debug_file, "email: $email\n");      // PII
fwrite($debug_file, "SQL Query: $insert\n"); // raw SQL + password hash
fwrite($debug_file, "Session set: uid=...\n"); // session data
After — all debug lines removed
// Remove all fopen / fwrite / fclose lines.
// Use php error_log() to server log if temporary debugging is needed.
// Never write to the web root.
Effect: User emails, SQL, and session data are no longer written to a publicly accessible file.
Critical New Finding

1.3 — Reset Tokens Have No Expiry — Old Links Work Forever

Done
New Findingreset-password-action.phploginClass.php — passupdate()

A reset token is stored with no timestamp. passupdate() only checks whether the token exists — never when it was issued. A link from months ago, found in an old email or browser history, still works.

Schema — add expiry column
ALTER TABLE users ADD COLUMN token_expires_at DATETIME NULL AFTER verify_token;
reset-password-action.php — set expiry + use secure token
$token   = bin2hex(random_bytes(16));               // replaces md5(rand())
$expires = date('Y-m-d H:i:s', strtotime('+1 hour'));
$stmt = $conn->prepare("UPDATE users SET verify_token=?, token_expires_at=? WHERE email=? LIMIT 1");
$stmt->bind_param('sss', $token, $expires, $email);
$stmt->execute();
loginClass passupdate() — check expiry before accepting
$row = $this->fetchOne('users', 'verify_token', $token);
if (!$row) { return 'Invalid token'; }
if (strtotime($row['token_expires_at']) < time()) {
    return 'This reset link has expired. Please request a new one.';
}
Effect: Reset links expire after 1 hour. Intercepted or old links are rejected automatically.

Security New Finding

2.1 — No CSRF Tokens on Login or Password Reset Forms

Done
New Findinglogin.phpreset-password.phploginaction.php

Neither form verifies that the POST came from the real page. A malicious site can auto-submit these forms on behalf of a visiting user.

login.php — generate token in session, embed in form
// Top of login.php (after session_start from 1.6):
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}
// In the form:
<input type="hidden" name="_token" value="<?= htmlspecialchars($_SESSION['csrf_token']) ?>">
loginaction.php — verify before any other processing
session_start();
if (!isset($_POST['_token']) || !hash_equals($_SESSION['csrf_token'], $_POST['_token'])) {
    echo json_encode(['status'=>'error','message'=>'Invalid request. Refresh and try again.']);
    exit;
}
Effect: Cross-site form submissions are rejected. hash_equals() prevents timing-based token comparison attacks.
Security New Finding

2.2 — Email Enumeration: Reset Endpoint Reveals Whether an Account Exists

Done
New Findingreset-password-action.php — lines 76–80

Returns "ERROR CHECK" when no account is found and "its available" when one is. An attacker can enumerate every registered email by submitting the form repeatedly.

Before
if(mysqli_num_rows($query_check) > 0){
    echo "its available";   // account exists
} else {
    echo "ERROR CHECK ";    // no account — leaks this fact
}
After — always same response
// Send the email silently if the account exists, do nothing if not.
// Client always sees the same message regardless.

Security Audit

2.3 — No Brute-Force Protection — Unlimited Password Guesses Allowed

Done
Audit ReportProject/login/loginClass.php — Login_test()

The original login() tracked attempts in login_attempt — it is now fully commented out. Login_test() has no rate limiting. Recommendation: lock per IP and per email after 5 failures in 15 minutes.

Recommended — check both IP and email before querying DB
private function countRecentFailures(string $col, string $val): int {
    $window = date('Y-m-d H:i:s', strtotime('-15 minutes'));
    $stmt   = $this->prepare("SELECT COUNT(*) FROM login_attempt WHERE `$col`=? AND attempted_at > ?");
    $stmt->bind_param('ss', $val, $window);
    $stmt->execute();
    return (int) $stmt->get_result()->fetch_row()[0];
}
// In login() — check before the user query:
if ($this->countRecentFailures('ip_address',$ip) >= 5 ||
    $this->countRecentFailures('email',$email)    >= 5) {
    return 'locked';
}
Effect: Password guessing is capped at 5 per 15 minutes by IP and by email. VPN-hopping does not bypass the email check.
Security Audit

2.4 — All SQL via Prepared Statements (Replace real_escape_string)

Done
Audit ReportARIA PlanProject/login/loginClass.php — all query methods

Every SQL query is built by concatenating escaped strings. Parameterized queries separate data from SQL structure at the protocol level — real_escape_string cannot achieve the same guarantee under all character encodings.

Before
$email = $this->real_escape_string($email);
$sql   = "SELECT * FROM users WHERE email = '$email'";
After — prepared statement (or BaseRepository fetchOne)
// Option A: direct prepared statement
$stmt = $this->prepare("SELECT * FROM users WHERE email = ?");
$stmt->bind_param("s", $email);
$stmt->execute();
$result = $stmt->get_result();

// Option B: after ARIA Phase 2 (Step 3 below)
$row = $this->fetchOne('users', 'email', $email);
Effect: SQL injection via any login field is impossible regardless of character encoding.
Security Audit

2.5 — No isset() Guards on $_POST in Handler Files

Done
Audit Reportloginaction.phpconfirmaction.phpreset-password-action.php

All three handlers read $_POST keys directly without isset(). A tampered request that omits a field causes PHP notices and passes null into the database layer.

Pattern — apply to every handler
if (!isset($_POST['email'], $_POST['password'])) {
    echo json_encode(['status'=>'error','message'=>'Missing required fields']);
    exit;
}
$email = $_POST['email'];
$pass  = $_POST['password'];
Effect: Malformed requests exit cleanly with a JSON error. No undefined variables reach the database layer.
Security Audit

2.6 — Weak Token Generation (md5(rand())) in Password Reset

Done
Audit Reportreset-password-action.phploginClass.php — passupdate() line 202

md5(rand()) is not cryptographically secure. rand() is seeded predictably, and MD5 is fast to brute-force. This is already covered by item 1.3 which uses bin2hex(random_bytes(16)).

Before
$token     = md5(rand())."funda"; // in reset-password-action.php
$new_token = md5(rand())."funda"; // in passupdate() after successful reset
After
$token     = bin2hex(random_bytes(16)); // 32-char cryptographically secure hex string
$new_token = bin2hex(random_bytes(16)); // rotate token after use
Effect: Reset tokens cannot be guessed or brute-forced with practical compute.

4.1 — Submit Button Has No Loading State — Double-Click Sends Two Requests

Done
New Findinglogin.php — AJAX block

The login button stays active during the AJAX call. Double-clicking fires two simultaneous POST requests, creating duplicate system_logs entries and potential race conditions on session creation.

After — disable on click, re-enable on response
var $btn = $(this).find('button[type="submit"]');

    ...

    }
});
Effect: Exactly one request per click. User gets instant visual confirmation their click registered.

BF-4 — confirmaction.php: Hard Session-uid Block Breaks Reset Flow

Fixed
confirmaction.php — lines 8–9

Before — hard block, reset flow always rejected

    Response::error('Session expired. Please log in again.');
}
$uid = $_SESSION['uid'];
After — uid from session OR token (either authorises the change)
/* uid from session (first-login) or empty string (reset — token identifies user) */

if (empty($uid) && empty($token)) {
    Response::error('Session expired. Please log in again.');
}
Effect: First-login flow (session uid set) and reset flow (token set, no session) both reach passupdate(). passupdate() identifies the user by token in both cases.
