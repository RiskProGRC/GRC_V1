Issue 3 — XSS Vulnerability in confirmpass.php (Unsanitised GET Parameters)

Pending Fix

File: confirmpass.php — lines 82, 86

$_GET["token"] and $_GET["email"] are printed directly into HTML without htmlspecialchars(). A crafted URL with a malicious value in either parameter will execute arbitrary JavaScript in the user's browser (Cross-Site Scripting).

Before (XSS vulnerable)
<input type="hidden" name="token"
    value="<?php if(isset($_GET["token"])){echo $_GET["token"];}?>">

<input type="email" name="email"
    value="<?php if(isset($_GET["email"])){echo $_GET["email"];}?>"
    class="form-control form-control-xl" readonly>
After (escaped output)
<input type="hidden" name="token"
    value="<?php if(isset($_GET["token"])){echo htmlspecialchars($_GET["token"], ENT_QUOTES, 'UTF-8');}?>">

<input type="email" name="email"
    value="<?php if(isset($_GET["email"])){echo htmlspecialchars($_GET["email"], ENT_QUOTES, 'UTF-8');}?>"
    class="form-control form-control-xl" readonly>
Effect: Prevents script injection via the password reset link. htmlspecialchars() converts <, >, ", and ' to safe HTML entities.

Issue 4 — confirmaction.php Trusts Client-Submitted uid (Hidden Field)

Pending Fix

Files: confirmpass.php — line 93   confirmaction.php — line 6

The logged-in user's ID is read from $_SESSION["uid"] in confirmpass.php, then written into a hidden <input> field and submitted back to confirmaction.php. confirmaction.php reads $_POST["uid"] directly. Any user can change the hidden field in browser developer tools and update another user's password.

Before — confirmpass.php (uid in hidden field)
<input type="hidden" name="uid" value="<?=$uid?>">
Before — confirmaction.php (trusts POST uid)
$uid   = $_POST["uid"];
$token = $_POST["token"];
$email = $_POST["email"];
$pass  = $_POST["pass"];
$hpass = password_hash($pass, PASSWORD_DEFAULT);
$alert = $loginclass->passupdate($uid, $hpass, $email, $token);
After — confirmaction.php (uid from session only)
session_start();
$uid   = $_SESSION["uid"] ?? null;   // never trust POST for uid
$token = $_POST["token"] ?? '';
$email = $_POST["email"] ?? '';
$pass  = $_POST["pass"]  ?? '';

if(!$uid || !$token || !$email || !$pass){ echo "Missing data"; exit; }

$hpass = password_hash($pass, PASSWORD_DEFAULT);
$alert = $loginclass->passupdate($uid, $hpass, $email, $token);
echo $alert;
Effect: The uid used to update the password is always taken from the server-side session, not from the form submission. A user cannot change another user's password by modifying the hidden field.

Issue 5 — No Brute-Force / Lockout Protection in Login_test()

Pending Fix

File: Project/login/loginClass.php — lines 235–280

The original login() method (now fully commented out) tracked failed login attempts in a login_attempt table and used $time_count to enforce a lockout. The replacement Login_test() has no rate limiting at all. An attacker can try unlimited passwords without any throttle or lockout.

Before (no protection)
public function Login_test($email, $pass, $ip_address){
    // ... password check only
    // no attempt counter, no lockout, no delay
}
After (recommended approach)
public function Login_test(string $email, string $pass, string $ip_address){
    // 1. Count failed attempts for this IP in the last 15 minutes
    $window = time() - 900;
    $count_sql = "SELECT COUNT(*) as attempts FROM login_attempt
                  WHERE ip_address='$ip_address' AND time_count > $window";
    $res = $this->query($count_sql);
    $row_c = $res->fetch_assoc();
    if((int)$row_c['attempts'] >= 5){
        return 'locked'; // new return code → show lockout message on UI
    }

    // 2. Normal credential check ...
    // 3. On failure, record the attempt:
    $t = time();
    $this->query("INSERT INTO login_attempt(ip_address,time_count)
                  VALUES('$ip_address','$t')");
    return 3;
}
Effect: After 5 failed attempts from the same IP within 15 minutes, the login endpoint returns a 'locked' response and the UI shows a lockout message instead of allowing further guesses.

Issue 6 — real_escape_string Used Instead of Prepared Statements

Pending Fix

File: Project/login/loginClass.php — all query methods

Every SQL query in loginClass.php is built by concatenating escaped strings. While real_escape_string reduces risk, it is not foolproof under all character encodings and is not the current PHP best practice. Parameterized queries with bind_param() guarantee the data is never interpreted as SQL regardless of encoding.

Before (string concatenation)
$email = $this->real_escape_string($email);
$sql   = "SELECT * FROM users WHERE email = '$email'";
After (prepared statement)
$stmt = $this->prepare("SELECT * FROM users WHERE email = ?");
$stmt->bind_param("s", $email);
$stmt->execute();
$result = $stmt->get_result();
Effect: Parameterized queries separate SQL logic from user data at the protocol level. SQL injection via email or password fields becomes impossible.

Issue 7 — No isset() Guards on $_POST in loginaction.php

Pending Fix

File: loginaction.php — lines 22–23

$email and $pass are read directly from $_POST without checking isset(). If the AJAX call is tampered with and those keys are missing, PHP throws an Undefined index notice. The undefined variables are then passed to Login_test() and compared against the database, which could produce unpredictable behaviour.

Before (no guard)
$email = $_POST["email"];
$pass  = $_POST["password"];
After (isset guard + early exit)
if(!isset($_POST['email'], $_POST['password'])){
    echo 3;
    exit;
}
$email = $_POST["email"];
$pass  = $_POST["password"];
Effect: Malformed requests that omit either field return response code 3 (invalid credentials) and exit cleanly — no PHP warnings, no undefined variable passed to the database layer.

Issue 9 — reset-password.php Always Shows "Success" Regardless of Outcome

Pending Fix

File: reset-password.php — lines 65–68

The AJAX success callback uses icon: "success" unconditionally. If reset-password-action.php returns "ERROR CHECK" (email not found), the user still sees a green success message. They are misled into thinking an email was sent when it was not.

Before (always success)
success:function(response){
    Swal.fire({
        icon: "success",         // always green — even on error
        title: response,
        timer: 500              // too short to read
    })
    window.setTimeout(function(){
        window.location.href="Login.php";
    }, 500);
}
After (handle success vs error)
success: function(response){
    response = $.trim(response);
    if(response === 'its available'){
        Swal.fire({
            icon: 'success',
            title: 'Email Sent',
            text: 'A password reset link has been sent to your inbox.',
            timer: 4000
        });
        setTimeout(function(){ window.location.href='login.php'; }, 4000);
    } else {
        Swal.fire({
            icon: 'error',
            title: 'Email Not Found',
            text: 'No account found with that email address.',
            confirmButtonText: 'Try Again'
        });
    }
}
Effect: Users who enter a wrong email address see a clear error. Only genuine success shows the green confirmation and redirects.

Issue 11 — No Show / Hide Password Toggle on Login Form

Pending Fix

File: login.php — lines 59–64

The password field has no eye icon to toggle visibility. Users who mistype their password cannot verify what they entered, which increases failed login attempts.

Change — add toggle icon + JS handler
<!-- HTML: add toggle span inside the password field wrapper -->
<span id="togglePassword" title="Show/hide password"
      style="position:absolute;right:15px;top:50%;transform:translateY(-50%);cursor:pointer;">
    <i class="bi bi-eye" id="eyeIcon"></i>
</span>

<!-- JS -->
$('#togglePassword').on('click', function(){
    var pwd  = $('#password');
    var icon = $('#eyeIcon');
    if(pwd.attr('type') === 'password'){
        pwd.attr('type', 'text');
        icon.removeClass('bi-eye').addClass('bi-eye-slash');
    } else {
        pwd.attr('type', 'password');
        icon.removeClass('bi-eye-slash').addClass('bi-eye');
    }
});
Effect: Users can reveal what they typed to confirm it before submitting, reducing failed logins caused by typos.
