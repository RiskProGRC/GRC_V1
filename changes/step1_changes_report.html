Bug Fix

Fix 1 — Validation Library Not Loading (layout/header.php)

✓ Fixed

File: layout/header.php — line 223

jquery.validate.js is a JavaScript library that was being included using a <link rel="stylesheet"> tag — the tag used for CSS files. Browsers silently discard it when loaded this way, so client-side form validation was never running on any page in the system.

Before
<!-- Validation -->
<link rel="stylesheet" href="../assets/vendors/validate/jquery.validate.js">
After
<!-- Validation -->
<script src="../assets/vendors/validate/jquery.validate.js"></script>
Effect: Client-side form validation now loads correctly on every page that uses jquery.validate rules.
Bug Fix

Fix 2 — Edit Entity Page Missing Session Guard (Project/entityEdit.php)

✓ Fixed

File: Project/entityEdit.php — line 14

All files inside Project/ use ../layout/header.php to include the shared header. entityEdit.php was missing the ../ prefix, so PHP could not find the file. The header silently failed to load — meaning the page had no navigation, no CSS, and most critically no session check. A user who was not logged in could access and submit this page without being redirected to login.

Before
<?php include_once("layout/header.php"); ?>
After
<?php include_once("../layout/header.php"); ?>
Effect: The Edit Entity page now loads with full navigation, styling, and session authentication enforced.
Bug Fix

Fix 3 — Role-Based Button Controls Not Applying (layout/footer.php)

✓ Fixed

File: layout/footer.php — lines 10–14

The footer echoes JavaScript that disables buttons based on the user's role (e.g. disabling add/edit/delete buttons for admin users on the permissions page). The jQuery calls were running inline — without waiting for the DOM to be fully ready — so the buttons they targeted may not have existed yet when the script ran, meaning the controls were never applied.

Before
<script>
$('.btn-userpermission-add').prop('disabled', true);
$('.btn-userpermission-delete').prop('disabled', true);
$('.btn-userpermission-edit').prop('disabled', true);
</script>
After
<script>
$(document).ready(function(){
    $('.btn-userpermission-add').prop('disabled', true);
    $('.btn-userpermission-delete').prop('disabled', true);
    $('.btn-userpermission-edit').prop('disabled', true);
});
</script>

Same fix applied to the Role 2 block that enables .btn-add-group.

Effect: Role-based button disabling now reliably applies after the DOM is fully loaded on every page.
What Has Not Changed

These fixes are surgical — no logic, no database queries, no AJAX handlers, and no page layouts were changed. Only the three specific broken lines were corrected. All other behaviour remains exactly as before.
