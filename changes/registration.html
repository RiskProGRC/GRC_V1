Accessibility Issues
1

Labels not linked to inputs

Critical

Every <label> in the form has no for attribute, and every <input> has no id. Screen readers cannot associate the label with its field — clicking a label also will not focus the input for mouse users.

Before
<label class="labels">Name</label>
<input type="text" class="form-control" name="fname" value="">

<label class="labels">Email ID</label>
<input type="text" class="form-control" name="email" value="">
After
<label class="labels" for="fname">Name</label>
<input type="text" id="fname" class="form-control" name="fname" value="">

<label class="labels" for="email">Email ID</label>
<input type="email" id="email" class="form-control" name="email" value="">
Apply to all fields: fname, sname, uname, phone, gender, email, roles, dept, password.
2

Empty id="" on all three <select> elements

Critical

id="" is invalid HTML. The three selects for Gender, User Roles, and Department all carry it. An empty id cannot be targeted by a for attribute, and multiple elements sharing an empty id violates the uniqueness constraint.

Before
<select class="form-select" name="gender" id="">
<select class="form-select" name="roles" id="">
<select class="form-select" name="dept" id="">
After
<select class="form-select" id="gender" name="gender">
<select class="form-select" id="roles" name="roles">
<select class="form-select" id="dept" name="dept">
3

Wrong input types on Email, Phone, and Password

High

Correct type values trigger the right keyboard on mobile, enable browser validation, and signal semantics to assistive technology. The password field has a capitalization error (type="Password") which technically works in most browsers but is non-standard and fragile.

Before
<input type="text"     name="phone"    ...>  <!-- should be tel -->
<input type="text"     name="email"    ...>  <!-- should be email -->
<input type="Password" name="password" ...>  <!-- wrong casing -->
After
<input type="tel"      id="phone"    name="phone"    ...>
<input type="email"    id="email"    name="email"    ...>
<input type="password" id="password" name="password" ...>
4

No aria-required or autocomplete on any field

High

aria-required="true" tells screen readers a field must be filled. autocomplete enables browser autofill and is required for WCAG 1.3.5 (Identify Input Purpose). Neither is present on any field.

After — full field set
<input type="text"     id="fname"    name="fname"    autocomplete="given-name"   aria-required="true">
<input type="text"     id="sname"    name="sname"    autocomplete="family-name"  aria-required="true">
<input type="text"     id="uname"    name="uname"    autocomplete="username"     aria-required="true">
<input type="tel"      id="phone"    name="phone"    autocomplete="tel"          aria-required="true">
<input type="email"    id="email"    name="email"    autocomplete="email"        aria-required="true">
<input type="password" id="password" name="password" autocomplete="new-password" aria-required="true">
5

No live region for submission feedback

Medium

When the form submits via AJAX there is nowhere to announce success or error to screen readers. A role="alert" region is announced automatically by assistive technology when its content changes.

Add before the </form> tag
<div id="form-feedback" role="alert" aria-live="polite" class="mt-3"></div>

Then in the AJAX success/error callback populate it:

$('#form-feedback').html('<div class="alert alert-success">User added successfully.</div>');
// or on error:
$('#form-feedback').html('<div class="alert alert-danger">' + res.message + '</div>');
Bugs
6

Stray PHP string delimiters '; bleeding into HTML

Critical

Lines 104 and 129 have '; after the closing </option> tag. These are remnants of a PHP echo statement that was refactored into inline HTML but not fully cleaned up. They render literally in the browser as visible characters.

Before (line 104)
        <option value="female">Female</option>';
After
        <option value="female">Female</option>
Same fix applies to line 129: <option value="2">User</option>';
7

$showdept and $profile are used but never fetched

Critical

The Department <select> loop on line 137 references $showdept and $profile, but neither variable is defined or fetched anywhere in this file. The page will throw a PHP warning and the department dropdown will be empty.

Current (broken)
<?php
include_once'./users/usersClass.php';
$usersclass = new usersClass();
// $showdept and $profile are never set
?>
Fix — fetch the data at the top of the file
<?php
include_once './users/usersClass.php';
$usersclass = new usersClass();
$showdept   = $usersclass->getDepartments();  // method must exist in usersClass
$profile    = [];                              // empty default — no pre-selected dept on add form
?>
8

Datatable script references #table1 which does not exist

Medium

The inline script at the bottom of the page initialises a Simple Datatable on #table1, but no table with that id exists on this form page. This throws a JS error on every page load and is likely copied from another page by mistake.

Before — remove this block entirely
<script src="../assets/vendors/simple-datatables/simple-datatables.js"></script>
<script>
    let table1 = document.querySelector('#table1');
    let dataTable = new simpleDatatables.DataTable(table1);
</script>
Architecture Gaps (ARIA Patterns)
9

No CSRF token on the form

Arch — ARIA Step 2

The CSRF pattern applied to the login system in Step 2 is missing here. Any authenticated user's browser could be tricked into posting a forged user-creation request.

Add to the form — PHP side
<?php if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
} ?>
<input type="hidden" name="csrf_token" value="<?= htmlspecialchars($_SESSION['csrf_token']) ?>">
Validate in the action handler
if (!hash_equals($_SESSION['csrf_token'], $_POST['csrf_token'] ?? '')) {
    Response::error('Invalid request.', 403);
}
10

ADD USERS button has no JS handler — form cannot submit

Arch — ARIA Steps 3 & 4

The button is type="button" (prevents native form submission) but no JavaScript is attached to it in this file or any included script. The form is effectively non-functional. A handler must be wired up following the ARIA Step 4 pattern: a dedicated JS file, dataType:'json', and the Response class on the server.

Handler skeleton — assets/js/usersadd.js (new file)
$('.adduser-button').on('click', function () {
    const $btn = $(this);
    $btn.prop('disabled', true).text('Adding…');
    $('#form-feedback').empty();

    $.ajax({
        url:      window.GRC.base + 'Project/usersaddaction.php',
        method:   'POST',
        dataType: 'json',
        data:     $('#profileform').serialize(),
        success: function (res) {
            if (res.status === 'success') {
                $('#form-feedback').html('<div class="alert alert-success">' + res.message + '</div>');
                $('#profileform')[0].reset();
            } else {
                $('#form-feedback').html('<div class="alert alert-danger">' + res.message + '</div>');
            }
        },
        error: function () {
            $('#form-feedback').html('<div class="alert alert-danger">Server error. Please try again.</div>');
        },
        complete: function () {
            $btn.prop('disabled', false).text('ADD USERS');
        }
    });
});
Server response — usersaddaction.php (new handler)
<?php
require_once '../Project/aria/Response.php';
require_once '../Project/aria/AuthGuard.php';
AuthGuard::require();

if (!hash_equals($_SESSION['csrf_token'], $_POST['csrf_token'] ?? '')) {
    Response::error('Invalid request.', 403);
}

require_once './users/usersClass.php';
$uc     = new usersClass();
$result = $uc->addUser($_POST);
$result ? Response::success('User added.') : Response::error('Could not add user.');
