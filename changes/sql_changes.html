Blocking — Run These First
Blocking

B1 — Add verify_token to users Table

Table: users   Used by: loginClass.php — resetPassword(), passupdate()   reset-password-action.php

The column verify_token is referenced in every part of the password reset flow. The UPDATE in reset-password-action.php writes to it; passupdate() queries it; confirmpass.php passes it via URL. Without this column every query silently fails or throws a MySQL error.

ALTER TABLE users ADD COLUMN verify_token VARCHAR(255) NULL DEFAULT NULL AFTER user_type;
Effect: Password reset token can now be written and read. The reset flow becomes functional for the first time.
Blocking

B2 — Add token_expires_at to users Table

Table: users   Used by: reset-password-action.php   loginClass.php — passupdate()

Step 1.3 code is already deployed. reset-password-action.php now saves this value on every token generation. passupdate() checks it before accepting a reset. Until this column exists the UPDATE in reset-password-action.php will throw a MySQL error on every reset attempt.

ALTER TABLE users ADD COLUMN token_expires_at DATETIME NULL DEFAULT NULL AFTER verify_token;

After running B1 and B2, the users table will have: ... user_type | verify_token | token_expires_at | bsc_setting ...

Effect: Reset tokens are now time-bounded to 1 hour. Old or intercepted reset links are rejected automatically.

Step 2 — Create Table

S1 — Create login_attempt Table

Table: login_attempt (does not exist yet)   Used by: master guide item 2.3

The brute-force protection in Step 2.3 tracks failed login attempts per IP and per email. The table does not currently exist. It needs both ip_address and email columns so the double-check (per-IP and per-email) works.

CREATE TABLE login_attempt ( id INT(11) NOT NULL AUTO_INCREMENT, ip_address VARCHAR(45) NOT NULL, email VARCHAR(255) NOT NULL DEFAULT '', attempted_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, PRIMARY KEY (id), INDEX idx_ip (ip_address, attempted_at), INDEX idx_email (email, attempted_at) ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

The two indexes mean the rate-limit queries run on indexed columns — no full table scans even with many rows. Old rows can be cleaned with a periodic job: DELETE FROM login_attempt WHERE attempted_at < DATE_SUB(NOW(), INTERVAL 1 DAY);

Effect: Step 2.3 brute-force protection becomes active. Failed attempts are tracked per IP and per email.
Existing Schema Issues — Not Blocking But Should Be Fixed
Schema Issue

E1 — users.phone Is INT(11) — Should Be VARCHAR(20)

Table: users   Current type: INT(11)   Correct type: VARCHAR(20)

Phone numbers are not integers. Storing them as INT causes three real problems:

  • Leading zeros are silently dropped — 0712345678 becomes 712345678
  • International format +254712345678 cannot be stored at all — the + causes a SQL error
  • The googlesignup() default of '0000' is stored as integer 0, not the string "0000"
-- Safe: VARCHAR stores the same digits, existing numeric values are preserved as strings ALTER TABLE users MODIFY COLUMN phone VARCHAR(20) NULL DEFAULT NULL;
Effect: Phone numbers with leading zeros and international formats can be stored correctly. No data loss on existing rows.
Schema Issue

E2 — users.bsc_setting Is YEAR(4) — Should Be TINYINT(1)

Table: users   Current type: YEAR(4)   Correct type: TINYINT(1)

The code sets bsc_setting = 0 as a flag (0 = not configured, 1 = configured). The YEAR type stores calendar years (1901–2155) — MySQL silently converts 0 to 0000 (an out-of-range sentinel value). Any code that compares bsc_setting == 0 may behave unexpectedly depending on how MySQL returns the value to PHP.

-- Current 0 values stored as YEAR '0000' become TINYINT 0 — logically equivalent ALTER TABLE users MODIFY COLUMN bsc_setting TINYINT(1) NOT NULL DEFAULT 0;
Effect: bsc_setting behaves as a clean boolean flag. No silent year-conversion side-effects.
Schema Issue

E3 — system_logs.entity Is VARCHAR(20) — Too Short

Table: system_logs   Current type: VARCHAR(20)   Correct type: VARCHAR(50)

The ActivityLogger::log() call passes module names as the $entity parameter — for example 'Login', 'Company', 'Risk Control'. At 20 characters, any entity name over that length is silently truncated, corrupting the audit log. 'Login' fits, but longer module names in future may not.

ALTER TABLE system_logs MODIFY COLUMN entity VARCHAR(50) NOT NULL;
Effect: Audit log entries are never silently truncated. All current values (under 20 chars) are unchanged.
