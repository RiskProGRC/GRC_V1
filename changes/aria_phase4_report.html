1 — The Problem

219 AJAX calls living inside PHP files

A scan of the entire Project/ directory found 182 inline $.ajax() calls across 52 PHP view files, plus the 37 calls already centralised in mine.js for the bussinf settings page.

Problem 1 — Not auditable. To find all AJAX targets you have to grep every PHP file. There is no single place to see what the frontend calls, what it sends, or what it expects back.
Problem 2 — Inconsistent patterns. Some calls use dataType:'text' and read a raw echo string. Others use dataType:"json" but with inconsistent success-handler shapes. Some use Toastify, some use Swal.fire, some use neither. There is no shared standard.
Problem 3 — PHP variables baked into JS. Inline scripts can reference <?=$suid?>, <?=$sdid?>, and getuseripaddress() directly in hidden form inputs that get serialised into AJAX data. A standalone JS file cannot do this — and after Phase 3 these values come from the server session anyway, so baking them in becomes a security concern.
Problem 4 — Nested event binding. Identified first in mine.js but present across many view files: a submit-handler registered inside a modal-open click handler stacks up every time the modal is opened, causing multiple AJAX fires per click.

2 — Approach

Two-part solution: config block + app.js

Part A — window.GRC config in header.php

layout/header.php already sets $suid, $sdid, $sess_roles, and calls getuseripaddress(). Add one small inline <script> block there (the only PHP-dependent JS that remains) to expose these as a global config object.

layout/header.php — add at end of <head> section
<script>
  window.GRC = {
    uid:   "<?= htmlspecialchars($suid, ENT_QUOTES) ?>",
    sdid:  "<?= htmlspecialchars($sdid, ENT_QUOTES) ?>",
    roles: <?= (int)$sess_roles ?>,
    ip:    "<?= htmlspecialchars(getuseripaddress(), ENT_QUOTES) ?>"
  };
</script>

After Phase 3 (AuthGuard), the PHP handlers get uid/sdid/ip from $_SESSION — they no longer trust values POSTed from the frontend. So window.GRC is only needed for UI logic (e.g. role-based show/hide of buttons), not for security-critical operations. This is a clean, safe pattern.

Part B — assets/js/app.js

One file, structured in sections by module. Every $.ajax() block that currently lives in a PHP view's <script> tag moves here. The PHP view loses its inline script entirely — it just includes the header (which loads app.js) and outputs HTML.

mine.js is merged in. The 37 calls in mine.js become the Settings section of app.js. The <script src="mine.js"> reference in bussinf.php is replaced by the shared app.js load in the header. One less file to maintain.

How app.js is loaded

Add one script tag at the bottom of layout/footer.php, after all vendor scripts:

layout/footer.php — add once
<script src="../assets/js/app.js"></script>

Because it loads on every page, the handlers are always available — delegated event listeners on document (the existing pattern) ensure they only fire when the matching element is present on the current page.

3 — How Phase 3 and Phase 4 interact

Phase 4 is most effective after Phase 3

ConcernBefore Phase 3After Phase 3 (ideal state)
uid / sdid sent to server Via hidden form inputs (<input type="hidden" name="entityuid" value="<?=$suid?>">) serialised into POST data Read from $_SESSION inside AuthGuard — frontend never needs to send them
Hidden inputs in forms Required — handlers use $_POST['entityuid'] Redundant — can be removed from HTML forms
IP address Called via getuseripaddress() inline in some hidden inputs Called inside AuthGuard on the PHP side
window.GRC.uid / .ip Still needed to populate hidden inputs Only needed for UI logic (role-based display) — not security

You can do Phase 4 before Phase 3 — it still works, since the hidden inputs still feed the handlers. But doing Phase 3 first means you can remove the hidden inputs from forms as you go, which shrinks the app.js data payloads and tightens security at the same time.

5 — Before / After code example

Example: approval AJAX in inbox.php

This shows a complete block — inline script in a PHP view — moved to app.js with the standardised JSON pattern.

Before — inside inbox.php <script>

$(document).on('click','.approverisk-btn',
function(e){
    e.preventDefault();
    $.ajax({
        url: 'riskapprove.php',
        method: 'post',
        data: $('#approveriskform').serialize(),
        dataType: 'json',
        success: function(r) {
            Toastify({
                text: r.message,

Recommended implementation order

  1. Complete Phase 3 (AuthGuard) first — removes the need for hidden uid/sdid inputs
  2. Create app.js skeleton with section headers and no code
  3. Add window.GRC to header.php and app.js load to footer.php
  4. Start with the settings sections (1–9): migrate mine.js + individual settings pages together — these are self-contained
  5. Move risk + control + entity sections next (10–12)
  6. Move KRI/BSC section (13–14)
  7. Move the remaining sections (15–19)
  8. Tackle inbox.php and riskstatus.php last — largest files, most calls
  9. Delete mine.js, remove bussinf.php script reference
End state: ~217 AJAX calls in one file, structured in 20 labelled sections (Section 20 is a comment placeholder for bulk_risk_upload.php). 50 of 52 PHP views are now plain HTML + PHP data rendering with no inline JavaScript. The 2 exceptions (bulk_risk_upload.php, risk_report.php) retain self-contained scripts that cannot be cleanly extracted. Debugging, auditing, and updating AJAX behaviour happens in one place.
