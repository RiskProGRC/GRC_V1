What Phase 1 Does

Phase 1 is purely additive. It creates a new Project/core/ folder containing three reusable base classes. No existing entity class, page file, or wrapper file is touched. Your application continues to run exactly as it does today. These classes are the foundation that Phases 2–5 will build on.

New File

1 — BaseRepository.php — Generic Data Access Layer

+ Security

Location: Project/core/BaseRepository.php

Every entity class (companyClass, processClass, recommendClass, etc.) contains the same fetch loop, the same single-row lookup, and the same delete pattern copy-pasted across every method. BaseRepository extracts all of these into reusable methods so entity classes never need to write them again.

Replaces these duplicate patterns across all entity classes:
  • The 10-line while($row = fetch_assoc()) loop in every showX() method
  • The single-row SELECT * WHERE id=X block in every fetchedit() / editDetails()
  • The filtered SELECT * WHERE col=val loop in every showXdept() method
  • The DELETE FROM table WHERE id=X block in every delete() method
  • The SELECT COUNT(*) pattern in every dashboard() method
Project/core/BaseRepository.php
<?php
require_once __DIR__ . '/../connection/connectionClass.php';

class BaseRepository extends connectionClass {

    protected function fetchAll(string $table): array {
        $stmt = $this->prepare("SELECT * FROM `$table`");
        $stmt->execute();
        return $stmt->get_result()->fetch_all(MYSQLI_ASSOC) ?? [];
    }

    protected function fetchWhere(string $table, string $col, $val): array {
        $stmt = $this->prepare("SELECT * FROM `$table` WHERE `$col` = ?");
        $stmt->bind_param('s', $val);
        $stmt->execute();
        return $stmt->get_result()->fetch_all(MYSQLI_ASSOC) ?? [];
    }

    protected function fetchOne(string $table, string $col, $val): ?array {
        $stmt = $this->prepare("SELECT * FROM `$table` WHERE `$col` = ? LIMIT 1");
        $stmt->bind_param('s', $val);
        $stmt->execute();
        return $stmt->get_result()->fetch_assoc() ?: null;
    }

    protected function deleteById(string $table, string $col, $val): bool {
        $stmt = $this->prepare("DELETE FROM `$table` WHERE `$col` = ?");
        $stmt->bind_param('s', $val);
        $stmt->execute();
        return $stmt->affected_rows > 0;
    }

    protected function countAll(string $table): int {
        $result = $this->query("SELECT COUNT(*) FROM `$table`");
        return (int) $result->fetch_row()[0];
    }

    protected function countWhere(string $table, string $col, $val): int {
        $stmt = $this->prepare("SELECT COUNT(*) FROM `$table` WHERE `$col` = ?");
        $stmt->bind_param('s', $val);
        $stmt->execute();
        return (int) $stmt->get_result()->fetch_row()[0];
    }
}
?>

Uses prepared statements throughout instead of real_escape_string. This is a free security upgrade — the existing entity classes use string interpolation which carries SQL injection risk even with escaping applied.

Used in Phase 2: Each entity class will change extends connectionClass to extends BaseRepository, then replace their duplicate methods with single calls like return $this->fetchAll('process');

2 — ActivityLogger.php — Centralised Audit Logging

Location: Project/core/ActivityLogger.php

Every write method in every entity class contains an identical 3-line block that inserts a row into system_logs. The only things that change are the entity name and action text. ActivityLogger is a static class that reduces this to a single readable line.

Replaces this block (copy-pasted ~40+ times across all entity classes):
  • companyClass — after every insert/update
  • processClass — after every insert/update
  • recommendClass — after every insert/update/approve/amend
  • incidentClass, actionClass, keyindicatorClass, and all others
Before — repeated in every write method
$insert = "INSERT INTO `system_logs`(`user_id`, `entity`, `action`, `ip_address`)
           VALUES ('$uid','Process','Added Process','$ipaddress')";
$query = $this->query($insert) or die($this->error);
Project/core/ActivityLogger.php
<?php
class ActivityLogger {

    public static function log(mysqli $conn, $uid, string $entity, string $action, string $ip): void {
        $stmt = $conn->prepare(
            "INSERT INTO `system_logs` (`user_id`, `entity`, `action`, `ip_address`) VALUES (?, ?, ?, ?)"
        );
        $stmt->bind_param('ssss', $uid, $entity, $action, $ip);
        $stmt->execute();
    }
}
?>
After — one line in any method that needs to log
ActivityLogger::log($this, $uid, 'Process', 'Added Process', $ipaddress);
Used in Phase 2: Every write method in every entity class replaces its 3-line logging block with the single ActivityLogger::log() call.

3 — Response.php — Standardised Output Utility

Location: Project/core/Response.php

Wrapper PHP files (e.g. processdelete.php, recommenddelete.php) all call a class method and echo a raw string back to AJAX. These strings are inconsistent — "DATA NOT INSERTED", "Record Deleted", "Process Deleted Sucessfully" (typo included). The JS side has no reliable way to tell success from failure. Response standardises every reply to a consistent JSON structure.

Project/core/Response.php
<?php
class Response {

    public static function success(string $message): void {
        header('Content-Type: application/json');
        echo json_encode(['status' => 'success', 'message' => $message]);
        exit;
    }

    public static function error(string $message): void {
        header('Content-Type: application/json');
        echo json_encode(['status' => 'error', 'message' => $message]);
        exit;
    }

    public static function redirect(string $url): void {
        header("Location: $url");
        exit;
    }
}
?>
Example — wrapper file before
$alert = $processclass->delete($pid);
echo $alert; // could be anything — "Process Deleted Sucessfully", "DATA NOT DELETED", etc.
Example — wrapper file after (Phase 4)
$ok = $processclass->deleteById('process', 'process_id', $pid);
$ok ? Response::success('Process deleted') : Response::error('Could not delete process');
Used in Phase 4: All AJAX wrapper files are standardised to use Response::success() / Response::error(), giving the JS side a reliable data.status to check.
