1. SQL Injection — Unsanitized POST Data in Queries

Affected files: Project/ajaxdata.php:3-6   Project/processfetch.php:7-9

Both files receive $_POST values and place them directly into SQL query strings with no escaping, no prepared statements, and no validation. These files are called by AJAX, making them directly reachable from the browser.

// ajaxdata.php — vulnerable
$did = $_POST['dept_id'];
$select = "SELECT process_id FROM dept_process WHERE dept_id='$did'";

// processfetch.php — vulnerable
$pid = $_POST["processid"];
$select = "SELECT * FROM risk WHERE process='$pid'";

Fix: Use prepared statements ($stmt = $conn->prepare(...)) or at minimum real_escape_string() on every POST value before using it in a query.

Impact: An attacker can read, modify, or delete any data in the database via a crafted AJAX request.

2. jQuery Loaded Twice on Almost Every Page

Location 1: layout/header.php:190 — loads jQuery 3.5.1 from Google CDN.

Location 2: ~20+ page files in Project/ (e.g. actions.php:348, addcontrol.php:231, activitylog.php:158) each load a second jQuery from assets/vendors/jquery/jquery.min.js.

Two different jQuery instances load and run on every one of these pages. This causes event handlers to bind twice, wastes bandwidth, and can cause unpredictable AJAX behaviour.

Fix: Remove the local jQuery <script> tags from every page file. The CDN version in header.php is sufficient.

Impact: Performance hit on every page load; risk of duplicate event handlers breaking AJAX calls.
High

3. Double Database Connection Opened in connect.php

File: Project/connection/connect.php:2-10

Every time connect.php is included it opens two separate connections to the same database — one procedural ($con) and one OOP ($db). Only one is ever needed.

$con = mysqli_connect("localhost","root","","grc"); // connection 1
$db  = new mysqli("localhost","root","","grc");      // connection 2 — redundant

Fix: Remove the $db line. Standardise the codebase to use either $con (procedural) or connectionClass (OOP) — not both.

Impact: Every page that includes connect.php opens an unnecessary second DB connection, consuming server resources.
High

4. JavaScript Validation File Included as CSS Stylesheet

File: layout/header.php:223

<link rel="stylesheet" href="../assets/vendors/validate/jquery.validate.js">

jquery.validate.js is a JavaScript library. Using <link rel="stylesheet"> to include it means the browser downloads it and discards it — the validation library never actually loads. Form validation is silently broken on every page.

Fix: Change to a <script src="..."></script> tag, placed after jQuery loads.

Impact: Client-side form validation does not work on any page in the system.
High

5. Wrong Include Path in entityEdit.php — Header Never Loads

File: Project/entityEdit.php:14

include_once("layout/header.php");  // WRONG — missing ../

All other files in Project/ use ../layout/header.php. This file is missing the ../, so the header silently fails to load on the Edit Entity page — no nav, no session check, no CSS.

Fix: Change to include_once("../layout/header.php");

Impact: The Edit Entity page renders without navigation or session authentication — users are not redirected to login even if they are not logged in.
Medium
Medium

6. Every Class Instantiated Twice — header.php + Page File

Affected files: layout/header.php:77-180 and pages like Project/risklist.php:1-12, Project/dashboard.php:3-42

header.php already creates instances of every major class (riskClass, controlClass, kiClass, incidentClass, recommendClass, departmentClass, etc.) and runs their queries. Individual page files then create the same classes again and run the same queries before including the header.

For example, risklist.php creates $riskClass = new riskClass() and calls showRisk() — then header.php creates another riskClass, and after the header loads, risklist.php calls showRisk() a second time via a different variable.

Fix: Move class instantiations to happen only in the header (or a dedicated bootstrap file) and reference those instances in page files, rather than re-creating them.

Impact: Every page executes the same database queries at least twice per load, doubling DB load unnecessarily.
Medium

7. 14 Files Bypass connectionClass with Direct DB Connections

The following files include connect.php directly, or create their own new mysqli(), instead of using connectionClass:

  • ajaxdata.php — creates new mysqli() directly with no include at all
  • dashboard.php, risk_register_report.php, risktreatment.php, entitynewaction.php, top10risk.php
  • bscimport.php, bsciso.php, check_db_structure.php, controllist.php
  • entityAction.php, processfetch.php, rcatlist.php, regexcel.php, registerAction.php

Fix: Standardise all DB access through connectionClass. If a file needs a raw connection, have it extend or instantiate connectionClass rather than opening a new one.

Impact: Credentials and connection logic are scattered. If the DB config changes, you must hunt through 14+ files to update it.
Medium

8. Hardcoded DB Credentials in Two Source Files

Files: Project/connection/connect.php   Project/connection/connectionClass.php

Both files contain hardcoded database credentials (host, username, password, database name), plus multiple commented-out sets of credentials for different environments (local dev, staging, production).

Fix: Move credentials to a single config.php file (excluded from git via .gitignore), or use environment variables via $_ENV / getenv(). Reference that config from both connection files.

Impact: Credentials checked into source control are a security risk. Multiple environment configs in comments makes deployment error-prone.
Medium

9. Footer JavaScript Runs Before jQuery and DOM Are Ready

File: layout/footer.php:3-15

The footer PHP echoes JavaScript that calls $('.btn-userpermission-add').prop('disabled', true). This inline script executes as the browser encounters it — at the footer position — before the page's <script> includes (jQuery, mine.js) have been parsed. The jQuery $ function is not yet defined, so the script throws an error and the buttons are never disabled.

Fix: Move this logic inside a $(document).ready() block, or place it after the JS includes at the very bottom of the page.

Impact: Permission-based UI controls (disabling buttons for certain roles) do not apply — users may see buttons they should not be able to click.
Medium

10. Same Role-Check Block Repeated 8 Times in header.php

File: layout/header.php:78-180

The pattern below is copy-pasted for every class — departments, processes, risks, controls, KPI, incidents, recommendations, and KRI:

if ($sess_roles == 1) {
    $showX = $xClass->showX();
} elseif ($sess_roles == 2) {
    $showX = $xClass->showXdept($sdid);
}

Also, the risk category block (lines 105-111) is identical for both roles — the same method is called either way, making the if/else pointless.

Fix: Create a helper function fetchByRole($class, $roleMethod, $deptMethod, $deptId) and call it once per class, replacing the 8 repeated blocks.

Impact: Adding a new role means updating the same pattern in 8 places — high risk of missing one.
Low
Low

11. Inline <style> Blocks Scattered Across 61 Page Files

Affected: 61 of 74 page files in Project/

Most page files contain a <style> block with rules like h4 { text-align: center; }, label { font-size: 15px; color: #000; }, and .form-control { font-size: 13px; } — the same declarations repeated across dozens of files.

Fix: Extract all common repeated rules into assets/css/app.css or assets/css/container.css. Keep page-specific overrides in a page-scoped class rather than a bare <style> block.

Impact: Style changes require editing dozens of files. Inconsistent styling across pages because each file has slightly different values.
Low

12. Hundreds of Lines of Commented-Out Dead Code in nav.php

File: layout/nav.php

Large blocks of HTML are commented out throughout nav.php — leftover examples from the Mazer UI template (breadcrumb, button, card, carousel, dropdown, chart links, support links, 3-level submenu examples). None of it is used or will be used.

Fix: Delete all commented-out blocks. The file will shrink from ~620 lines to roughly 150 lines of actual navigation markup.

Impact: Hard to read and maintain. Increases the chance of accidentally un-commenting the wrong block.
Low

13. Debug and Test Files Present in the Production Folder

The following files appear to be development artifacts with no production purpose:

  • Project/login_test.php
  • Project/check_db_structure.php
  • Project/test.php
  • Project/blank.php
  • Project/settings/blank.php

Fix: Confirm none of these are referenced by other files, then delete them. Verify check_db_structure.php is not reachable publicly — it likely exposes DB schema information.

Impact: check_db_structure.php in particular may expose database schema to anyone who navigates to it directly.
