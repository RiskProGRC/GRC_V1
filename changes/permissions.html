Implementation Approach

CSS body-class technique

A role-readonly class is added to <body> when the logged-in user's role is 1 or 3. A CSS rule then sets opacity: 0.4, pointer-events: none, and cursor: not-allowed on every element carrying btn-userpermission-edit, btn-userpermission-delete, or btn-userpermission-add.

This approach is DataTable-safe — because the rule lives on the body element, it re-applies automatically after DataTables redraws rows during search or pagination without any extra JavaScript.

<!-- Body tag (PHP) --> <body class="<?php echo (isset($sess_roles) && in_array((int)$sess_roles, [1, 3])) ? 'role-readonly' : '' ?>"> /* CSS rule added to each page */ .role-readonly .btn-userpermission-edit, .role-readonly .btn-userpermission-delete, .role-readonly .btn-userpermission-add { opacity: 0.4; pointer-events: none; cursor: not-allowed; }

Inbox — Department Filter for Role 3  (2026-07-01)

What changed

When the logged-in user has roles = 3, Project/inbox.php now fetches only records that belong to their department ($_SESSION['dept_id']). All five tabs are scoped: Risks, Controls, KPIs, Recommendations, and Actions. The tab badge counts (pending / approved / rejected) are also recalculated from the filtered dataset so the numbers match what is displayed in the table.

For all other roles (admin, manager, champion) the existing global fetch methods run unchanged, so behaviour is identical to before for non-role-3 users.

// read session role and department — role 3 is a department-level user if (session_status() === PHP_SESSION_NONE) { session_start(); } $sessionRole = $_SESSION['roles'] ?? 0; $sessionDept = $_SESSION['dept_id'] ?? ''; if ($sessionRole == 3 && $sessionDept !== '') { // dept-filtered data $showRisk = $riskClass->showRiskdept($sessionDept); $showcontrol = $controlClass->showcontroldept($sessionDept); $showki = $kiclass->showKidept($sessionDept); $showrecommend = $recommendClass->showrecommenddept($sessionDept); $showaction = $actionClass->showactiondept($sessionDept); // counts derived from filtered arrays — no extra DB calls $pending_risk = count(array_filter($showRisk, fn($r) => $r['approval'] == 1)); // ... same pattern for all other counts } else { // admin / other roles — fetch everything $showRisk = $riskClass->showRisk(); $pending_risk = $riskClass->pendingrisks(); // ... }

Risk Tracker (riskview.php) — Dept Dropdown Scoped for Roles 2 & 3  (2026-07-01)

What changed

The Choose Entity dropdown in Project/riskview.php previously listed every department regardless of who was logged in. For roles 2 and 3, the dropdown now shows only the department the user belongs to ($_SESSION['dept_id']). The select is also disabled so they cannot switch to another department.

Additionally, for roles 2 and 3 the risk register loads automatically on page open via a $(document).ready call to loadDept() — no manual selection is required.

Role 1 (Administrator) is unaffected — they still see the full department list and choose freely.

KRI (kri.php) — Fetch Only Department Records for Roles 2 & 3  (2026-07-01)

What changed

Project/kri.php previously fetched all rows from the performance table via fetchkri(). For roles 2 and 3, it now calls the new fetchkrideptperf($dept) method which JOINs performance with risk and filters by risk.dept = $sessionDept.

A JOIN is required because the performance table has no dept_id column — it links to a department only through its risk_id foreign key. Role 1 (Administrator) is unaffected and still sees all rows.
