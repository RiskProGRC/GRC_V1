1 — No Session Regeneration After Login (Session Fixation)

Pending

File: Project/login/loginClass.php — Login_test() lines 250–253

Attack scenario: An attacker tricks a victim into loading login.php?PHPSESSID=abc123. The victim logs in. Because the session ID is never rotated, the attacker — who already knows the ID abc123 — now has full authenticated access to the victim's account.

After a successful password check, four $_SESSION keys are set but session_regenerate_id() is never called. The same session ID that existed before the user logged in remains in use after they are authenticated. This is the textbook definition of a session fixation vulnerability.

Before — session ID unchanged at login
if (password_verify($pass, $row['upassword']) && $row["user_type"] == 1) {
    $_SESSION["uid"]     = $row["id"];
    $_SESSION["user"]    = $row["username"];
    $_SESSION["dept_id"] = $row["dept_id"];
    $_SESSION["roles"]   = $row["roles"];
    // ← session ID is still the pre-login ID
After — one line eliminates session fixation
if (password_verify($pass, $row['upassword']) && $row["user_type"] == 1) {
    session_regenerate_id(true); // ← issues a new session ID, destroys the old one
    $_SESSION["uid"]     = $row["id"];
    $_SESSION["user"]    = $row["username"];
    $_SESSION["dept_id"] = $row["dept_id"];
    $_SESSION["roles"]   = $row["roles"];
Why previous reports missed this: Both reports focused on what was in the session (correct keys, correct values). Neither asked whether the session ID itself was being rotated at the authentication boundary.
Effect: Any session ID an attacker knew before login becomes invalid the moment the user authenticates. Session fixation is fully closed.

2 — Debug Log Writes SQL + Emails to a Public File on Every Google Login

Pending

File: Project/login/loginClass.php — googlesignup() lines 58–163

Risk: The file googlesignup_debug.log is created in the web root and is publicly readable at http://<your-domain>/GRC_V1/googlesignup_debug.log. It contains every Google user's email address, the full INSERT SQL including the hashed password, and session data. This is a live data breach.

The method opens a log file on line 59, then writes to it at least 5 times per call — every time a user signs in with Google. This was debugging code that was never cleaned up before going to production.

Before — 15+ lines writing private data to a public file
$debug_file = fopen("googlesignup_debug.log", "a");
fwrite($debug_file, "googlesignup called ... for email: $email\n"); // PII
fwrite($debug_file, "Checking if user exists with email: $email\n"); // PII
// ... inside the INSERT branch:
fwrite($debug_file, "SQL Query: $insert\n");    // raw SQL + hashed password
fwrite($debug_file, "Session set: uid=...\n"); // session data
fwrite($debug_file, "Redirecting to: ...\n");
fclose($debug_file);
After — all debug lines removed
public function googlesignup(string $name, string $email, string $password): void {
    // No file logging. Use php error_log() (server-side only) if temporary
    // debugging is ever needed again — never write to the web root.
    ...
Why previous reports missed this: The audit report scanned login.php, loginaction.php, and the reset flow. The googlesignup() method is a separate code path only triggered by the Google OAuth callback — it was not reviewed as part of the main login audit.
Effect: User emails, SQL, and session data are no longer written to a publicly accessible file. The method is safe to call in production.

3 — Password Reset Tokens Have No Expiry — Old Links Work Forever

Pending

Files: reset-password-action.php   Project/login/loginClass.php — passupdate()

Attack scenario: A user requests a reset in January, doesn't use the link, and deletes the email. In June, the email is accessed by someone else (shared inbox, phishing, data breach). The January reset link still works. The attacker resets the password and takes over the account.

reset-password-action.php saves a token with no timestamp. passupdate() only checks WHERE verify_token='$token' — no time check. Tokens are valid indefinitely. After a successful reset the old token is replaced with md5(rand())."funda" (also weak) but old tokens from abandoned reset requests are never cleared.

Recommended schema change
ALTER TABLE users
    ADD COLUMN token_expires_at DATETIME NULL AFTER verify_token;
reset-password-action.php — save token with expiry + use secure token
$token   = bin2hex(random_bytes(16));               // replaces md5(rand())
$expires = date('Y-m-d H:i:s', strtotime('+1 hour')); // 1-hour window

$stmt = $conn->prepare(
    "UPDATE users SET verify_token=?, token_expires_at=? WHERE email=? LIMIT 1"
);
$stmt->bind_param('sss', $token, $expires, $email);
$stmt->execute();
loginClass passupdate() — check expiry before accepting token
$row = $this->fetchOne('users', 'verify_token', $token);

if (!$row) {
    return 'Invalid token';
}
if (strtotime($row['token_expires_at']) < time()) {
    return 'This reset link has expired. Please request a new one.';
}
// only reach here if token exists AND has not expired
Why previous reports missed this: Both reports noted the weak md5(rand()) token generation but neither checked whether a timestamp was stored or validated. The token flow was reviewed for correctness, not for time-bounding.
Effect: Reset links expire after 1 hour. Old links from previous requests (abandoned, leaked, or stolen) are rejected automatically.

4 — No CSRF Tokens on Login or Password Reset Forms

Pending

Files: login.php   reset-password.php   loginaction.php   reset-password-action.php

Neither the login form nor the reset form include a CSRF token. A malicious page on any other domain can construct a hidden form that POSTs to these endpoints and auto-submit it when a user visits. Without a token, the server cannot tell whether the request came from the real form or an attacker's page.

login.php — generate token in session + embed in form
session_start();
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}
...
<form id="loginform">
    <input type="hidden" name="_token" value="<?= htmlspecialchars($_SESSION['csrf_token']) ?>">
loginaction.php — verify token before any other processing
session_start();
if (
    !isset($_POST['_token']) ||
    !hash_equals($_SESSION['csrf_token'], $_POST['_token'])
) {
    Response::error('Invalid request. Please refresh and try again.');
}
Why previous reports missed this: The audit report checked what happens after submission (SQL injection, session handling). The ARIA plan focused on prepared statements and the Response class. Neither checked whether the request origin was validated before processing started.
Effect: Cross-site form submissions are rejected. hash_equals() prevents timing-based token comparison attacks.

5 — Reset Form Reveals Whether an Email Address Has an Account

Pending

File: reset-password-action.php — lines 76–80

The reset endpoint returns two visibly different responses: "ERROR CHECK" when the email has no account, and "its available" when it does. An attacker can submit any email address and immediately know whether that person has a GRC account. This is called email enumeration and allows targeted phishing of confirmed users.

Before — different responses leak account existence
if (mysqli_num_rows($query_check) > 0) {
    // ... send email ...
    echo "its available";   // account exists
} else {
    echo "ERROR CHECK ";    // no account — reveals this
}
After — identical response either way
// Send the email only if the account exists (silently skip if not).
// The client always receives the same response — attacker learns nothing.
Response::success('If that email is registered, a reset link has been sent.');
Why previous reports missed this: Both reports noted the inconsistent response text as a grammar/UX issue ("its available" lacks an apostrophe) but neither flagged the security implication of using different response values for found vs. not-found.
Effect: An attacker cannot determine from the response whether an email address is registered. The message is truthful either way.

6 — Submit Button Has No Loading State — Double-Click Sends Two Requests

Pending

File: login.php — AJAX block

The login button is not disabled during the AJAX call. A user who double-clicks, or clicks once and then clicks again because nothing appeared to happen, fires two simultaneous POST requests to loginaction.php. Both will create a system_logs entry and the second may arrive while the first is still setting session variables. The button also gives no visual feedback that the click was received.

Before — button stays active during request
$("#loginform").submit(function(e) {
    e.preventDefault();
    $.ajax({
        url: "loginaction.php",
        ...
        // button is never disabled — user can click multiple times
    });
});
After — disable on click, re-enable on completion
$("#loginform").submit(function(e) {
    e.preventDefault();
    var $btn = $(this).find('button[type="submit"]');
    $btn.prop('disabled', true).text('Logging in...');

    $.ajax({
        url: "loginaction.php",
        ...
        complete: function() {
            $btn.prop('disabled', false).text('Log In'); // always re-enable
        }
    });
});
Effect: Exactly one request per click. User sees instant confirmation their click registered. Duplicate system_log entries from double-clicks are prevented.

7 — Login Fields Have No autocomplete Attributes

Pending

File: login.php — lines 54 and 60

The email field uses type="text" (not type="email") and neither field has an autocomplete attribute. As a result:

  • Password managers may fail to fill in credentials correctly or not recognise the fields at all
  • Mobile browsers show a plain keyboard instead of an email-optimised one for the email field
  • Browser autofill may guess wrong and fill the username into the email field
Before
<input type="text"     name="email"    placeholder="Email"    class="...">
<input type="password" name="password" placeholder="Password" class="...">
After
<input type="email"    name="email"    placeholder="Email"    autocomplete="email"            class="...">
<input type="password" name="password" placeholder="Password" autocomplete="current-password"  class="...">
Effect: Password managers work correctly. Mobile shows the right keyboard. Browsers fill the right fields. type="email" adds client-side format checking for free.

8 — systemlog() Has a Dead Branch: count < 0 Can Never Be True

Pending

File: Project/login/loginClass.php — systemlog() lines 340–357

mysqli_result::num_rows returns an integer that is always ≥ 0. The condition if ($count < 0) can never evaluate to true. The return "NULL VALUES" path is permanently dead code. The method also uses a while loop + $rows[] to build an array when a single fetchAll() call achieves the same result in one line (after ARIA Phase 2).

Before — unreachable branch + manual array building
Public function systemlog(){
    $select = "SELECT * FROM system_logs";
    $query  = $this->query($select) or die($this->error);
    $count  = $query->num_rows;

    if ($count < 0) {              // ← NEVER TRUE
        return "NULL VALUES";      // ← dead code
    } else {
        while ($row = $query->fetch_assoc()) {
            $rows[] = $row;        // manual array building
        }
        if (!empty($rows)) {
            return $rows;
        } else {
            return array();
        }
    }
}
After — clean one-liner using BaseRepository
public function systemlog(): array {
    return $this->fetchAll('system_logs'); // returns [] when empty, never null
}
Why previous reports missed this: The audit report listed systemlog() as using a raw query (correct) but did not analyse the logic of the conditional. The ARIA plan recommended converting the method to fetchAll() without noting that the existing conditional was already broken.
Effect: Dead code removed. Return type is predictable — always an array, never "NULL VALUES" string. Consistent with all other list methods in the system.
