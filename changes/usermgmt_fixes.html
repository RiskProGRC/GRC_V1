Fix 1 — Session
Duplicate session_start() notice on every page
layout/header.php — line 2
Critical

Symptom: Clicking "User Management" in the nav produced a PHP notice: "session_start(): Ignoring session_start() because a session is already active in layout/header.php on line 2".

Root cause: The new User Management pages (userslist.php, usersadd.php, profile.php) start the session early (so the CSRF token can be generated before header.php is included). header.php then unconditionally called session_start() a second time, which PHP rejects.

Before
<?php
session_start();
After
<?php
if (session_status() === PHP_SESSION_NONE) {
    session_start();
}
This single change fixes the notice on every page in the project, because header.php is included by all authenticated views.
Fix 2 — Data Bug
Suspended count always showed 0 on the stats dashboard
Project/users/usersClass.php — getStats() + addusers()
Data Bug

Symptom: The "Suspended" stat card on userslist.php always displayed 0 even when suspended users existed.

Root cause: Two compounding problems:

  • The addusers() INSERT did not include the access column, so every new user was stored with access = NULL.
  • The getStats() query used SUM(access = 0) which returns 0 for NULL rows — MySQL treats NULL = 0 as NULL, not true.
  • The user list view showed NULL access users as "Suspended" (because the condition was access == 1 ? 'Active' : 'Suspended'), so the list and the count were inconsistent.

Fix 1 — addusers() INSERT:

/* Before */
"INSERT INTO users(fname,sname,gender,dept_id,email,phone,username,upassword,roles,user_type)
 VALUES(?,?,?,?,?,?,?,?,?,?)"

/* After — access defaults to 1 (Active) for every new user */
"INSERT INTO users(fname,sname,gender,dept_id,email,phone,username,upassword,roles,user_type,access)
 VALUES(?,?,?,?,?,?,?,?,?,?,1)"

Fix 2 — getStats() query:

/* Before */
SUM(access = 1) AS active,
SUM(access = 0) AS suspended,   -- misses NULL rows

/* After — suspended = everyone who is not explicitly active */
COALESCE(SUM(access = 1), 0)                       AS active,
COUNT(*) - COALESCE(SUM(access = 1), 0)          AS suspended,
Existing users in the database with access = NULL will now be counted as Suspended, matching what the list view already showed. Run a one-time update if you want to mark them all Active: UPDATE users SET access = 1 WHERE access IS NULL AND deleted_at IS NULL;

assets/js/app.js — line 452
Critical

Symptom: Clicking ADD USER (even with empty fields) showed a SweetAlert error dialog saying "FORM ERROR", then the page navigated to userslist.php.

Root cause: app.js (loaded globally via footer.php) still contained an old .adduser-button click handler with no validation. When the button was clicked:

Fix 5 — Form Bug
Pressing Enter in any field navigated away from the page
Project/usersadd.php + assets/js/usermanagement.js
Form Bug

Symptom: Pressing Enter inside any text input submitted the form natively, reloading the page and discarding any entered data.

Root cause: The <form> tag had no onsubmit handler. Browsers submit any form on Enter by default, regardless of button type="button" attributes.

Fix 1 — HTML (inline guard, runs before JS):

<form id="profileform" onsubmit="return false;">

Fix 2 — JS (belt-and-suspenders):

$('#profileform').on('submit', function (e) { e.preventDefault(); });
Fix 6 — Form Bug
Empty form could still POST if GRC.validate was unavailable + correct success/error navigation
assets/js/usermanagement.js — AJAX block
Form Bug

Two improvements were applied to the AJAX submit block:

1 — beforeSend guard (belt-and-suspenders POST prevention):
jQuery cancels the request entirely if beforeSend returns false. Even if GRC.validate somehow returns true incorrectly, the request cannot fire with empty required fields.

beforeSend: function () {
    var empty = addUserRules.some(function (r) {
        return !$('#' + r.field).val().trim();
    });
    if (empty) {
        $btn.prop('disabled', false).text('ADD USER');
        showFeedback('Please fill in all required fields.', 'warning');
        return false; // cancels the jQuery AJAX request
    }
},

2 — Correct post-submit navigation:

  • Success → show green message for 1.5 s, then redirect to userslist.php.
  • Server error (e.g. duplicate email, FORM ERROR) → show red inline message, re-enable button, stay on usersadd.php so the user can correct and resubmit.
success: function (res) {
    if (res.status === 'ok') {
        showFeedback(res.message, 'success');
        setTimeout(function () {
            window.location.href = base + 'Project/userslist.php'; // redirect on success
        }, 1500);
    } else {
        showFeedback(res.message, 'danger');
        $btn.prop('disabled', false).text('ADD USER'); // stay on page on error
    }
},
