Problem — What the Scan Found

Three Critical Issues in Every Handler File

Vulnerability 1 — No authentication gate. Every AJAX handler file (actionadd.php, riskassessaction.php, etc.) can be called directly by anyone who knows the URL. There is no session check. An unauthenticated user can POST to these files and write to the database.
Vulnerability 2 — User ID comes from $_POST, not the session. Every handler reads $uid = $_POST["uid"]. A malicious user can submit any uid they like, making every ActivityLogger entry and every record insert attributable to a forged identity. The form sends uid as a hidden HTML field, which is trivially editable in browser dev tools.
Vulnerability 3 — IP address is forged by the client. The form sends $_POST["ip"] populated from a JavaScript call to getuseripaddress(). Any client can submit a fake IP. Server-side IP must be read from $_SERVER['REMOTE_ADDR'].
Code quality — include_once vs require_once. All handler files use include_once for their entity class. If the class file is missing (wrong path, deployment error), PHP silently continues and the handler crashes with an undefined-class error instead of failing immediately and clearly.
Consistency — plain text responses. All handlers echo raw strings ("Action Added Successfully", "ENTER process data", etc.). The JavaScript reads these as unstructured text and places them directly into SweetAlert titles. A response with quotes, HTML, or a PHP warning bleeds into the UI. A structured JSON envelope fixes this.

New File — AuthGuard.php

Project/core/AuthGuard.php

One file required at the top of every handler. It does four things: starts the session, validates it, exposes safe variables, and provides the two response helpers.

Project/core/AuthGuard.php
<?php
session_start();

// Reject unauthenticated requests immediately.
if (!isset($_SESSION['uid'], $_SESSION['user'])) {
    http_response_code(401);
    echo json_encode(['status' => 'error', 'message' => 'Unauthorised']);
    exit;
}

// Safe variables available to every handler.
$uid  = (string) $_SESSION['uid'];
$sdid = (string) ($_SESSION['dept_id'] ?? '');
$ip   = _guard_ip();

function _guard_ip(): string {
    if (!empty($_SERVER['HTTP_CLIENT_IP'])) {
        return $_SERVER['HTTP_CLIENT_IP'];
    }
    if (!empty($_SERVER['HTTP_X_FORWARDED_FOR'])) {
        return explode(',', $_SERVER['HTTP_X_FORWARDED_FOR'])[0];
    }
    return $_SERVER['REMOTE_ADDR'] ?? '0.0.0.0';
}

// JSON response helpers used by all handler files.
function respond_ok(string $message): never {
    header('Content-Type: application/json');
    echo json_encode(['status' => 'ok', 'message' => $message]);
    exit;
}
function respond_error(string $message): never {
    header('Content-Type: application/json');
    echo json_encode(['status' => 'error', 'message' => $message]);
    exit;
}

Frontend — AJAX Response Update

JavaScript Before / After

Because handlers now return JSON, the success callback in every AJAX call must read response.message instead of the raw response string. This is the only change needed on the frontend side.

Before — JS in form page
$.ajax({
  url: 'actionadd.php',
  method: 'post',
  data: $("#form").serialize(),
  dataType: 'text',
  success: function(response) {
    Swal.fire({
      icon: 'success',
      title: response, // raw string
      timer: 1500
    });
  }
});
After — JS in form page
$.ajax({
  url: 'actionadd.php',
  method: 'post',
  data: $("#form").serialize(),
  dataType: 'json',
  success: function(response) {
    var icon = response.status === 'ok'
                ? 'success' : 'error';
    Swal.fire({
      icon: icon,
      title: response.message, // from JSON
      timer: 1500
    });
  }
});
Also: remove the two hidden form fields <input type="hidden" name="uid"> and <input type="hidden" name="ip"> from every view page — they are no longer sent and are a security exposure.

Bug found — nested event handler accumulation

Every modal-open click re-registers its inner submit handler. Clicking .addcs (add control strength) opens the modal and registers a new click handler on .addcs-btn. Click the open button 3 times, then submit — the AJAX fires 3 times. This affects every section that uses the modal-open pattern: process, users, impact, likelihood, risk category, control type, and control strength (7 sections).
Before — handler stacks up on every open
// Outer click opens modal AND registers inner handler

    $('#addcs-modal').modal('show');

    // NEW handler bound every time modal opens ↓

Summary

Phase 3 in One Paragraph

Phase 3 introduces a single new file — AuthGuard.php — and makes it the first line of every AJAX handler in the project. The guard starts the session, validates it, and exposes $uid, $sdid, and $ip from server-side session data. Every handler's include_once becomes require_once. Every echo $string response becomes respond_ok() or respond_error() returning a JSON envelope. On the frontend, each AJAX call switches dataType to 'json' and reads response.message instead of the raw string. Hidden uid and ip form fields are removed from all view pages. The result: an unauthenticated request is rejected before it reaches any business logic, user identity cannot be forged, and the response contract between frontend and backend is consistent and machine-readable.
