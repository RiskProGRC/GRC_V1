Overview

The original riskstatus.php + entityAction.php pattern works by POSTing a department ID via jQuery AJAX, and entityAction.php returns a raw HTML blob which jQuery injects directly into the page. This creates tight coupling between PHP and HTML, makes columns names hard to verify, and hides SQL errors silently.

riskview.php + riskview_fetch.php rebuilds this using a JSON API pattern: the fetch endpoint returns structured data, and JavaScript renders every table client-side. This separates concerns, surfaces errors cleanly, and makes the queries independently testable.

Architecture

Request Flow

Design
// 1. User selects department from dropdown in riskview.php
deptSelect.addEventListener('change', () => {
    fetch('riskview_fetch.php', { method: 'POST', body: 'deptid=5' })
    .then(r => r.json())
    .then(data => {
        renderRisks(data.risks);       // fills #tab-risks
        renderKI(data.ki);             // fills #tab-ki
        renderControls(data.controls); // fills #tab-controls
        renderActions(data.actions);   // fills #tab-actions
        renderRecommendations(data.recommendations);
    });
});

Each render function builds a full <table> string from the JSON array and injects it into the matching tab pane. SimpleDatatables is initialized after injection.

Bugs Found & Fixed During Build

4 schema issues discovered

Bug 1 — Wrong column name: control vs controls

Schema

The text description column in the control table is named controls (plural). All existing queries — including entityAction.php — were using control.control which doesn't exist. Controls tab was silently returning no rows.

Wrong:SELECT control.control FROM control
Fixed:SELECT `control`.controls FROM `control`

Bug 2 — Non-existent column: control.risk

Schema

The control table has no direct FK to the risk table. The link is through the risk_control junction table (control_id → risk_id). Any query doing JOIN risk ON control.risk = risk.risk_id returns no rows.

Wrong:FROM control INNER JOIN risk ON control.risk = risk.risk_id
Fixed:FROM `control` INNER JOIN risk_control rcl ON `control`.control_id = rcl.control_id INNER JOIN risk r ON rcl.risk_id = r.risk_id

Bug 3 — Ambiguous dept_id in controls & actions WHERE clause

SQL

After adding INNER JOIN process to the controls and actions queries, MySQL could not resolve WHERE dept_id = ? because both control and process tables have a dept_id column.

Wrong:WHERE dept_id = ?
Fixed:WHERE `control`.dept_id = ?  /  WHERE action.dept_id = ?

Bug 4 — HTML response instead of JSON (stray output)

PHP

PHP warnings or notices output before header('Content-Type: application/json') caused the fetch() to receive HTML, making JSON.parse() throw "Unexpected token '<'". Fixed with ob_start() at the top of the fetch file and mysqli_report(MYSQLI_REPORT_ERROR | MYSQLI_REPORT_STRICT) to surface real errors.

ob_start();                          // capture everything from includes
require_once 'AuthGuard.php';
include_once 'connect.php';
$stray = trim(ob_get_clean());      // collect leaked output
if ($stray) { json_error($stray); }  // return it as readable JSON

riskview_fetch.php — All 6 Queries

Prepared statements, verified columns

Controls Query — Most Complex

6 JOINs

Requires the junction table risk_control to link controls back to risks. All 6 tables joined in one statement.

SELECT `control`.control_id, `control`.controls,
       r.risk_id, r.risk_name, p.process_name,
       cs.cs_name, ct.ct_name,
       CONCAT(u.fname,' ',u.sname) AS reviewer_name
FROM `control`
INNER JOIN risk_control     rcl ON `control`.control_id = rcl.control_id
INNER JOIN risk             r   ON rcl.risk_id         = r.risk_id
INNER JOIN control_strength cs  ON `control`.cstrength = cs.strength_id
INNER JOIN control_type     ct  ON `control`.ctype     = ct.ctype_id
INNER JOIN process          p   ON `control`.process_id= p.process_id
INNER JOIN users            u   ON `control`.reviewer  = u.id
WHERE `control`.dept_id = ?
ORDER BY r.risk_id ASC

Risks Query

2 JOINs
SELECT r.risk_id, r.risk_name, r.cause, r.nominee, r.rdate,
       rc.name AS category, CONCAT(u.fname,' ',u.sname) AS reviewer_name
FROM risk r
INNER JOIN riskcat rc ON r.rcat     = rc.riskcat_id
INNER JOIN users   u  ON r.reviewer = u.id
WHERE r.dept = ?   -- note: column is 'dept' not 'dept_id'
