Day 1 — Foundation & Security Layer

Goal: understand why the old code was insecure and how the new core files fix it.

1

Foundation & Security Layer

The audit, the core classes, and one handler traced end to end

Morning Read the audit report 30 min

Open changes/grc_audit_report.html and read findings 1–6. This is the "before" picture — what was broken, why it mattered, and what the plan was to fix it. Every subsequent change traces back to one of these findings.

  • changes/grc_audit_report.html — read findings 1–6
  • changes/step1_changes_report.html through step5_changes_report.html — skim each (5 min each); these were the quick pre-ARIA fixes
Mid-morning Read the three core files 45 min

Read these in order — each one depends on the previous. By the end you should be able to explain what every variable in AuthGuard.php is, where it comes from, and what the two response helpers output.

  • Project/connection/connectionClass.php — the OOP mysqli connection; provides $db
  • Project/core/BaseRepository.php — the prepared-statement wrapper; all entity classes extend this
  • Project/core/AuthGuard.php — 30 lines; sets $uid, $sdid, $ip from session; defines respond_ok() and respond_error()
Key question to answer: Before Phase 3, where did $uid come from in a handler like actionadd.php? After Phase 3, where does it come from? Why does the difference matter?
Afternoon Trace one entity class — before vs after 45 min

Pick action/actionClass.php. Read it alongside changes/aria_phase2_report.html. Focus on: where is the SQL now? Where does the audit log write happen? What does $this->prepare() resolve to?

  • changes/aria_phase2_report.html — read the before/after section
  • Project/action/actionClass.php — read the full class
  • Project/core/BaseRepository.php — re-read prepare(), fetchAll(), fetchOne()
End of day Confirm one handler matches the pattern 30 min

Open the Phase 3 report and read the actionadd.php before/after example. Then open the real file and confirm it matches the "after" pattern. Check: is $_POST["uid"] gone? Is respond_ok() used? Does AuthGuard load first?

  • changes/aria_phase3_report.html — read the before/after code example
  • Project/actionadd.php — verify against the report
  • Project/core/AuthGuard.php — re-read once more with context
Day 2 — Trace a Full Request End to End

Goal: follow one user action from browser click all the way to the database and back.

2

Full Request Trace

Browser → app.js → handler → entity class → database → JSON → browser

Morning Trace: adding a KRI entry 60 min

Follow this exact sequence through the code. At each step, write down one sentence describing what that layer's job is.

  • assets/js/app.js — find Section 13 (KRI). Read the .addkri click handler. Find grcSwal at the top of the file and read what it does with r.status and r.message.
  • Project/kriaction.php — confirm: no $_POST["uid"]. Trace where $uid comes from.
  • Project/raf/kriClass.php — find addkrientry(). What does $this->prepare() resolve to?
  • Project/core/AuthGuard.php — follow respond_ok($msg). What JSON shape reaches the browser?
Key question: What happens if the user's session has expired and they click "Add KRI"? Trace the code path — at which line does execution stop, and what does the browser receive?
Mid-morning Trace: adding a company (settings flow) 30 min

Repeat the same trace for a simpler settings operation. This reinforces the same pattern from a different angle — notice how the structure is identical even though the domain is different.

  • assets/js/app.js — find Section 1. Read the .addcompany click handler.
  • Project/companyaction.php — read the full handler.
  • Project/company/companyClass.php — find addCompany().
Afternoon Read Phase 3 and Phase 4 reports back to back 60 min

Now that you've traced real code, the "why" in each report will be concrete rather than abstract. Pay attention to how Phase 3 (JSON responses) and Phase 4 (app.js) were designed to work together — the handlers produce {"status":"ok","message":"..."} and grcSwal(r) consumes it.

  • changes/aria_phase3_report.html — read in full, focusing on the request flow diagram
  • changes/aria_phase4_report.html — read in full, focusing on the before/after code example and the section structure table
Notice: Phase 4 was implemented before Phase 3 was complete. This is why the JS side (grcSwal, dataType:'json') was already aligned — the frontend was written to expect the JSON format that Phase 3 then delivered on the backend.
Day 3 — Frontend & The Big Picture

Goal: understand how app.js loads, how view files changed, and consolidate everything into one mental model.

3

Frontend & Big Picture

How app.js loads, how view files changed, and drawing the architecture from memory

Morning How app.js loads on every page 30 min

Understand the two wires that connect the PHP session to the JavaScript layer. Every authenticated page runs through these two files.

  • layout/header.php — find the window.GRC = { uid, sdid, roles, ip } block. This is where PHP session data becomes a JS global.
  • layout/footer.php — find the app.js script tag. This is the single load point for all 20 sections.
  • assets/js/app.js — read just the top ~30 lines: the grcSwal() and grcToast() helpers and the global datepicker init block.
Key question: If you needed to add a new AJAX handler for a new feature, where exactly would you add it? What section number would it get?
Mid-morning Before vs after a view file 45 min

Open a view file that was stripped in Phase 4. It now has no <script> block. Find its section in app.js and understand what was moved where.

  • Project/actions.php — confirm: no inline <script> block at the bottom
  • assets/js/app.js — find Section 15 (Actions). Read each handler and match it to the buttons/links in actions.php.
  • changes/aria_phase4_report.html — read the "How PHP views change" before/after example (Section 5 of the report)
Afternoon Draw the architecture from memory 45 min

Without looking at any file, sketch the full request path on paper. Then open the files to verify. If you can draw this and explain each layer's job, you understand the full refactor.

Browser click └── app.js handler // Section N — delegated $(document).on() └── $.ajax POST // url: 'handler.php', dataType: 'json' └── handler.php └── require AuthGuard // session gate → $uid, $ip, $sdid └── EntityClass::method() └── BaseRepository::prepare() └── MySQLi prepared statement └── ActivityLogger::log() // audit trail └── respond_ok($msg) // {"status":"ok","message":"..."} └── grcSwal(r) / grcToast(r) // reads r.status + r.message
You're done when you can answer: (1) Why can't a user forge their own uid anymore? (2) Why does a SQL injection on a handler input fail silently? (3) If a session expires mid-session, what does the user see and why?
