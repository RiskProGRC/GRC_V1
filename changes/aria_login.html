Change 2 — Login_test() uses fetchOne() + ActivityLogger + returns status strings

Pending

File: Project/login/loginClass.php — lines 235–280

The current method uses a raw SELECT * string query with real_escape_string. It also has an inline INSERT INTO system_logs. Both are replaced. The method is also renamed from Login_test to login, and the return values change from 2/3/text to predictable status strings.

Before
public function Login_test($email, $pass, $ip_address){
    $email = $this->real_escape_string($email);
    $sql   = "SELECT * FROM users WHERE email = '$email'";
    $result = $this->query($sql);
    ...
    $insert = "INSERT INTO `system_logs`(`user_id`, `entity`, `action`, `ip_address`)
               VALUES ('$userlogid','Login','user $user Logged in','$ip_address')";
    $this->query($insert);

    return 'WELCOME &nbsp;&nbsp;' . $_SESSION["user"]; // success
    return 2;  // first login
    return 3;  // wrong credentials
}
After
public function login(string $email, string $pass, string $ip_address): string {
    $row = $this->fetchOne('users', 'email', $email); // prepared statement lookup

    if (!$row || !password_verify($pass, $row['upassword'])) {
        return 'error';        // wrong email or password — same response (security)
    }
    if ((int)$row['access'] === 2) {
        return 'suspended';    // account disabled by admin
    }

    $_SESSION['uid']     = $row['id'];
    $_SESSION['user']    = $row['username'];
    $_SESSION['dept_id'] = $row['dept_id'];
    $_SESSION['roles']   = $row['roles'];

    if ((int)$row['user_type'] === 0) {
        return 'first_login';  // must change password before using the system
    }

    ActivityLogger::log($this, $row['id'], 'Login', 'user ' . $row['username'] . ' Logged in', $ip_address);
    return 'success:' . $row['username'];
}
Effect: No raw SQL in the login path. Audit log uses the central ActivityLogger. Return values are readable strings instead of magic numbers 2 and 3.

Change 3 — passupdate() uses fetchOne() + prepared statements, removes echo

Pending

File: Project/login/loginClass.php — lines 182–232

Currently uses a raw SELECT with real_escape_string and has echo inside the method body instead of return. Also uses the weak md5(rand()) to generate a replacement token.

Before
$check_token = "SELECT * FROM users WHERE verify_token='$token' LIMIT 1";
$query_token = $this->query($check_token) or die($this->error);
...
$new_token = md5(rand()) . "funda";
...
echo "Missing Fields available"; // should be return
echo "No tokens available";      // should be return
After
public function passupdate(string $hpass, string $email, string $token): string {
    if (empty($token) || empty($email) || empty($hpass)) {
        return 'Missing required fields'; // return not echo
    }

    $row = $this->fetchOne('users', 'verify_token', $token); // prepared lookup
    if (!$row) { return 'Invalid or expired reset token'; }

    $stmt = $this->prepare("UPDATE users SET upassword=?, user_type=1 WHERE verify_token=?");
    $stmt->bind_param('ss', $hpass, $token);
    $stmt->execute();
    if ($stmt->affected_rows < 1) { return 'Password not updated'; }

    $new_token = bin2hex(random_bytes(16)); // cryptographically secure token
    $stmt2 = $this->prepare("UPDATE users SET verify_token=? WHERE verify_token=?");
    $stmt2->bind_param('ss', $new_token, $token);
    $stmt2->execute();
    return 'Password updated';
}
Effect: No raw SQL, no echo inside a class method, cryptographically secure token invalidation. The $uid parameter is removed — the token alone is sufficient for identity.

Change 6 — loginaction.php uses Response class + isset() guard + JSON output

Pending

File: loginaction.php — all lines

Currently reads $_POST without checking isset(), calls a method named login_test, and echoes raw text (2, 3, or a string). Phase 3 replaces this with Response::success() / Response::error() so the frontend always receives consistent JSON.

Note: AuthGuard.php is NOT used here — the user is not yet authenticated when this file runs. session_start() is added manually instead.

Before
$email = $_POST["email"];   // no isset() guard
$pass  = $_POST["password"];
$alert = $loginclass->login_test($email, $pass, $ip_address);
echo $alert;  // raw text: "WELCOME ...", 2, or 3
After
require_once './Project/core/Response.php';
require_once './Project/login/loginClass.php';
session_start();

if (!isset($_POST['email'], $_POST['password'])) {
    Response::error('Missing credentials');
}

$result = (new loginClass())->login($_POST['email'], $_POST['password'], getClientIp());

match($result) {
    'first_login' => Response::success('first_login'),
    'suspended'   => Response::error('Your account is suspended. Contact the administrator.'),
    'error'       => Response::error('Invalid email or password. Please try again.'),
    default       => Response::success(substr($result, 8)) // strip 'success:' prefix
};
Effect: All login responses are now JSON {"status":"...","message":"..."}. The frontend switches from dataType:'text' to dataType:'json' and checks response.status.

After

if (!isset($_SESSION['uid'])) {
    Response::error('Session expired. Please log in again.');
}
if (!isset($_POST['token'], $_POST['email'], $_POST['pass'])) {
    Response::error('Missing required fields.');

$hpass  = password_hash($_POST['pass'], PASSWORD_DEFAULT);

$result === 'Password updated'
    ? Response::success($result)
    : Response::error($result);

Change 12 — confirmpass.php fixes XSS on GET values

Pending

File: confirmpass.php — lines 82, 86

$_GET["token"] and $_GET["email"] are echoed directly into HTML attributes — a Cross-Site Scripting vulnerability. Any malicious value in the URL injects into the page.

Before (XSS vulnerable)
value="<?php if(isset($_GET["token"])){echo $_GET["token"];}?>"
value="<?php if(isset($_GET["email"])){echo $_GET["email"];}?>"
After (safe output)
value="<?php if(isset($_GET["token"])){echo htmlspecialchars($_GET["token"], ENT_QUOTES,'UTF-8');}?>"
value="<?php if(isset($_GET["email"])){echo htmlspecialchars($_GET["email"], ENT_QUOTES,'UTF-8');}?>"
Effect: Script injection via the password-reset URL is blocked. htmlspecialchars() encodes <, >, ", and ' before they reach the HTML output.
